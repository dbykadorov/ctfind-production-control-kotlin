# Feature Specification: Edit Existing Users in Cabinet

**Feature Branch**: `014-edit-users`  
**Created**: 2026-04-30  
**Status**: Accepted  
**Input**: User description: "Редактирование существующих пользователей в кабинете для роли ADMIN"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Admin Updates User Access (Priority: P1)

Администратор открывает существующего пользователя и меняет отображаемое имя и набор ролей, чтобы быстро корректировать доступ без создания нового аккаунта.

**Why this priority**: Это основная бизнес-ценность фичи: управление доступом должно исправляться сразу в рабочем процессе.

**Independent Test**: Администратор редактирует пользователя из списка, сохраняет изменения и видит обновленные данные в списке и поиске.

**Acceptance Scenarios**:

1. **Given** администратор находится в разделе пользователей и выбирает пользователя, **When** он меняет отображаемое имя и сохраняет, **Then** новое имя отображается в списке пользователей.
2. **Given** администратор редактирует пользователя, **When** он обновляет набор ролей и сохраняет, **Then** новый набор ролей отображается в списке пользователей и применяется для дальнейшего доступа пользователя.

---

### User Story 2 - Security Guardrails for Editing (Priority: P1)

Система разрешает редактирование только администраторам и не позволяет нарушить минимальные правила доступа.

**Why this priority**: Без этих ограничений фича может привести к эскалации прав или потере административного контроля.

**Independent Test**: Не-администратор пытается редактировать пользователя и получает отказ; попытка снять административную роль у последнего администратора также отклоняется.

**Acceptance Scenarios**:

1. **Given** пользователь без административных прав, **When** он пытается выполнить редактирование пользователя, **Then** система отклоняет действие.
2. **Given** в системе остался один активный администратор, **When** выполняется попытка убрать у него административную роль, **Then** система отклоняет сохранение и сообщает причину.

---

### User Story 3 - Transparent Edit Outcome (Priority: P2)

После редактирования администратор получает понятный результат операции, а изменения остаются прослеживаемыми для контроля.

**Why this priority**: Ясная обратная связь и трассируемость уменьшают операционные ошибки и ускоряют диагностику.

**Independent Test**: Успешное и неуспешное редактирование показывают понятные сообщения пользователю; успешные изменения отражаются в журнале аудита без чувствительных данных.

**Acceptance Scenarios**:

1. **Given** администратор успешно сохраняет изменения пользователя, **When** операция завершена, **Then** система показывает подтверждение успеха и обновляет список.
2. **Given** администратор сохраняет невалидные изменения, **When** операция отклонена, **Then** система показывает понятную ошибку и не изменяет данные пользователя.

---

### Edge Cases

- Редактируется пользователь, который уже был удален или недоступен к моменту сохранения.
- Передано пустое отображаемое имя.
- Передан пустой набор ролей.
- Переданы неизвестные или повторяющиеся роли.
- Происходит одновременное редактирование одного и того же пользователя двумя администраторами.
- Администратор пытается снять административную роль у последнего активного администратора.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST позволять администратору открыть режим редактирования для существующего пользователя из списка пользователей.
- **FR-002**: Система MUST позволять изменять только отображаемое имя пользователя и набор его ролей.
- **FR-003**: Система MUST отображать логин пользователя в режиме редактирования только для просмотра, без возможности изменения.
- **FR-004**: Система MUST требовать непустое отображаемое имя при сохранении изменений.
- **FR-005**: Система MUST требовать непустой набор ролей при сохранении изменений.
- **FR-006**: Система MUST принимать только роли из поддерживаемого каталога ролей.
- **FR-007**: Система MUST нормализовать повторяющиеся роли как единый набор ролей пользователя.
- **FR-008**: Система MUST отклонять попытки редактирования для пользователей без административных прав.
- **FR-009**: Система MUST отклонять попытки редактирования для неаутентифицированных пользователей.
- **FR-010**: Система MUST отклонять сохранение, если целевой пользователь не существует на момент операции.
- **FR-011**: Система MUST предотвращать удаление административной роли у последнего активного администратора.
- **FR-012**: Система MUST сохранять изменения так, чтобы они были видны в списке пользователей сразу после успешного завершения операции.
- **FR-013**: Система MUST показывать понятный результат операции: подтверждение успеха при сохранении и объяснимую причину при отказе.
- **FR-014**: Система MUST записывать успешные изменения пользователя в аудит с указанием кто изменил, кого изменили и какие поля были изменены.
- **FR-015**: Система MUST исключать чувствительные данные (пароли и хеши паролей) из сообщений интерфейса, результатов операции и аудита.
- **FR-016**: Система MUST не включать в данный инкремент изменение логина, изменение пароля, блокировку, удаление и массовое редактирование пользователей.

### Key Entities *(include if feature involves data)*

- **User Account Profile**: учетная запись пользователя с идентификатором, логином, отображаемым именем, признаком активности и назначенными ролями.
- **Role Assignment Set**: нормализованный набор ролей пользователя, определяющий его доступ к разделам кабинета.
- **User Update Audit Record**: запись трассировки успешного изменения пользователя, включающая инициатора, цель и измененные атрибуты без чувствительных данных.

### Constitution Alignment *(mandatory)*

- **ERP/domain fit**: Усиливает операционный контур управления доступом к заказам, производству, складу и административным разделам в рамках Phase 01.
- **TOC readiness**: Поддерживает корректное распределение ролей по рабочим зонам, что снижает задержки от неверного доступа и ручных обходных операций.
- **Traceability/audit**: Все успешные изменения профиля пользователя фиксируются как бизнес-события с явной связью "кто изменил / кого изменил / что изменил".
- **Security/API boundary**: Редактирование ограничено администраторами; неаутентифицированные и неавторизованные попытки отклоняются; действует защита от потери последнего администратора.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 100% проверок администратор может изменить отображаемое имя и роли существующего пользователя и увидеть обновление в списке сразу после сохранения.
- **SC-002**: В 100% проверок пользователь без административных прав не может изменить данные другого пользователя.
- **SC-003**: В 100% проверок попытка снять административную роль у последнего активного администратора блокируется.
- **SC-004**: В 100% успешных изменений создается аудит-запись без включения пароля или хеша пароля.
- **SC-005**: Не менее 95% ручных операций редактирования завершаются менее чем за 1 минуту от открытия формы до подтверждения результата.

## Assumptions

- Поддерживаемый каталог ролей Phase 01 остается неизменным в рамках этого инкремента.
- Редактирование доступно только из существующего раздела пользователей, без отдельного экрана профиля.
- Администратор может редактировать собственную учетную запись при соблюдении правила "не удалять последнего администратора".
- Смена логина и пароля остается отдельным будущим инкрементом.
