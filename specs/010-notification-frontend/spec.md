# Feature Specification: Фронтенд уведомлений — колокольчик, dropdown, страница

**Feature Branch**: `010-notification-frontend`
**Created**: 2026-04-29
**Status**: Draft
**Input**: M7 Спека 3 из 3 — Frontend UI для внутренних уведомлений: badge с count непрочитанных в TopBar, dropdown со списком, страница /cabinet/notifications с пагинацией, навигация по клику.

## User Scenarios & Testing

### User Story 1 — Badge непрочитанных в TopBar (Priority: P1)

Любой аутентифицированный пользователь видит в шапке приложения иконку колокольчика. Рядом с колокольчиком отображается числовой badge с количеством непрочитанных уведомлений. Badge обновляется автоматически каждые 30 секунд через polling. Если непрочитанных нет — badge скрыт, колокольчик остаётся видимым.

**Why this priority**: Без badge пользователь не знает о новых уведомлениях. Это минимальный индикатор, без которого вся система уведомлений бесполезна на фронте.

**Independent Test**: Открыть приложение, создать уведомление через backend, подождать до 30 секунд — badge появляется с числом. Прочитать все — badge исчезает.

**Acceptance Scenarios**:

1. **Given** пользователь аутентифицирован и имеет 3 непрочитанных уведомления, **When** страница загружена, **Then** рядом с колокольчиком отображается badge «3».
2. **Given** пользователь не имеет непрочитанных уведомлений, **When** страница загружена, **Then** колокольчик отображается без badge.
3. **Given** badge показывает «2», **When** приходит новое уведомление и проходит polling-цикл (30 с), **Then** badge обновляется до «3».
4. **Given** badge показывает «5», **When** пользователь переходит на другую страницу приложения, **Then** badge остаётся видимым и актуальным (polling продолжается).

---

### User Story 2 — Dropdown со списком уведомлений (Priority: P1)

По клику на колокольчик открывается выпадающий список (dropdown/popover) с последними уведомлениями (до 10 штук). Каждое уведомление показывает: иконку типа, заголовок (title), относительное время («5 мин назад»). Непрочитанные визуально выделены. По клику на уведомление — оно отмечается прочитанным и пользователь переходит на карточку связанного объекта (задача или заказ). В заголовке dropdown есть кнопка «Отметить все прочитанными». Внизу dropdown — ссылка «Все уведомления» ведущая на полную страницу.

**Why this priority**: Dropdown — основной способ взаимодействия с уведомлениями. Без него badge бесполезен — пользователь видит число, но не может узнать о чём уведомления.

**Independent Test**: Кликнуть на колокольчик → dropdown показывает список. Кликнуть на уведомление → переход на карточку, уведомление прочитано. Нажать «Отметить все» → все помечены, badge исчезает.

**Acceptance Scenarios**:

1. **Given** есть 5 непрочитанных уведомлений, **When** пользователь кликает на колокольчик, **Then** открывается dropdown с 5 уведомлениями, непрочитанные визуально выделены.
2. **Given** dropdown открыт и показывает уведомление типа TASK_ASSIGNED с targetType=PRODUCTION_TASK, **When** пользователь кликает на него, **Then** уведомление помечается прочитанным и пользователь переходит на карточку производственной задачи.
3. **Given** dropdown открыт, есть 3 непрочитанных, **When** пользователь нажимает «Отметить все прочитанными», **Then** все уведомления помечаются прочитанными, badge скрывается.
4. **Given** dropdown открыт, **When** пользователь кликает «Все уведомления», **Then** происходит переход на /cabinet/notifications.
5. **Given** уведомлений нет, **When** пользователь кликает на колокольчик, **Then** dropdown показывает пустое состояние «Нет уведомлений».
6. **Given** dropdown открыт, **When** пользователь кликает вне dropdown, **Then** dropdown закрывается.

### User Story 3 — Страница уведомлений с пагинацией (Priority: P2)

Полноценная страница /cabinet/notifications со списком всех уведомлений текущего пользователя. Поддерживает пагинацию (20 на страницу), фильтр «только непрочитанные», кнопку «Отметить все прочитанными». Каждое уведомление кликабельно — переход на карточку объекта. Непрочитанные визуально выделены.

**Why this priority**: Страница нужна для просмотра полной истории уведомлений, но большинство повседневных взаимодействий происходит через dropdown.

**Independent Test**: Перейти на /cabinet/notifications → список уведомлений с пагинацией. Включить фильтр «только непрочитанные» → показываются только непрочитанные. Кликнуть на уведомление → переход на карточку.

**Acceptance Scenarios**:

1. **Given** у пользователя 25 уведомлений, **When** он переходит на /cabinet/notifications, **Then** показываются первые 20 уведомлений с навигацией на вторую страницу.
2. **Given** пользователь на странице уведомлений, **When** он включает фильтр «только непрочитанные», **Then** показываются только непрочитанные, пагинация пересчитывается.
3. **Given** пользователь на странице уведомлений, **When** он нажимает «Отметить все прочитанными», **Then** все уведомления помечаются прочитанными, badge в TopBar обновляется.
4. **Given** пользователь на странице уведомлений, **When** он кликает на уведомление с targetType=ORDER, **Then** происходит переход на карточку заказа.

### User Story 4 — Пункт в Sidebar (Priority: P2)

В боковой панели навигации добавлен пункт «Уведомления» (иконка Bell), доступный всем аутентифицированным ролям. Пункт ведёт на /cabinet/notifications.

**Why this priority**: Sidebar — стандартный способ навигации в приложении. Пользователи ожидают, что все основные разделы доступны оттуда.

**Independent Test**: Открыть sidebar → виден пункт «Уведомления». Кликнуть → переход на /cabinet/notifications.

**Acceptance Scenarios**:

1. **Given** пользователь с любой ролью (ADMIN, ORDER_MANAGER, PRODUCTION_SUPERVISOR, PRODUCTION_EXECUTOR), **When** он открывает sidebar, **Then** пункт «Уведомления» виден.
2. **Given** пользователь на любой странице, **When** он кликает «Уведомления» в sidebar, **Then** происходит переход на /cabinet/notifications с правильным выделением активного пункта.

---

### Edge Cases

- Что происходит при клике на уведомление без targetType/targetId (системное уведомление)? Уведомление помечается прочитанным, но навигация не происходит.
- Что происходит, если объект по targetId не найден (задача удалена)? Пользователь переходит на страницу объекта, которая покажет свой стандартный 404.
- Как badge ведёт себя при потере сети? Polling прекращается, badge показывает последнее известное значение. При восстановлении — polling возобновляется.
- Что если количество непрочитанных > 99? Badge показывает «99+».
- Как dropdown ведёт себя на мобильной ширине? Dropdown адаптируется к ширине экрана (полная ширина на узких экранах).

## Requirements

### Functional Requirements

- **FR-001**: Система MUST отображать иконку колокольчика в правой части TopBar для всех аутентифицированных пользователей.
- **FR-002**: Система MUST показывать числовой badge с количеством непрочитанных уведомлений рядом с колокольчиком. При 0 непрочитанных badge скрыт. При > 99 отображается «99+».
- **FR-003**: Система MUST обновлять count непрочитанных каждые 30 секунд через polling endpoint GET /api/notifications/unread-count.
- **FR-004**: Система MUST открывать dropdown/popover по клику на колокольчик, показывающий до 10 последних уведомлений.
- **FR-005**: Каждое уведомление в dropdown MUST показывать: иконку по типу, заголовок (title), относительное время.
- **FR-006**: Непрочитанные уведомления MUST быть визуально выделены (цвет фона или точка).
- **FR-007**: По клику на уведомление с targetType и targetId система MUST пометить его прочитанным (PATCH /api/notifications/{id}/read) и перенаправить на карточку объекта.
- **FR-008**: Dropdown MUST содержать кнопку «Отметить все прочитанными» (POST /api/notifications/mark-all-read).
- **FR-009**: Dropdown MUST содержать ссылку «Все уведомления», ведущую на /cabinet/notifications.
- **FR-010**: Страница /cabinet/notifications MUST показывать полный список уведомлений с пагинацией (20 на страницу, GET /api/notifications с page/size).
- **FR-011**: Страница уведомлений MUST поддерживать фильтр «только непрочитанные» (параметр unreadOnly=true).
- **FR-012**: Пункт «Уведомления» MUST быть добавлен в Sidebar, доступен всем аутентифицированным ролям.
- **FR-013**: Навигация по клику на уведомление MUST маппить targetType на маршруты: PRODUCTION_TASK → /cabinet/production-tasks/:id, ORDER → /cabinet/orders/:id.
- **FR-014**: Для навигации по targetId (человекочитаемый номер) к карточке объекта (UUID маршрут) backend MUST предоставить UUID объекта в ответе уведомления (новое поле targetEntityId в NotificationResponse).

### Key Entities

- **Notification (frontend)**: Представление уведомления на клиенте — id, type, title, body, targetType, targetId, targetEntityId, read, readAt, createdAt. Маппится из NotificationResponse backend API.
- **NotificationType**: Визуальная категория — определяет иконку и цвет (TASK_ASSIGNED → иконка назначения, STATUS_CHANGED → иконка статуса, TASK_OVERDUE → иконка предупреждения).

### Constitution Alignment

- **ERP/domain fit**: Уведомления — оперативный канал обратной связи в производственном контроле. Фронтенд замыкает цикл: backend генерирует уведомления по бизнес-событиям → фронтенд доставляет их пользователю → пользователь реагирует.
- **TOC readiness**: Уведомления о просрочке (TASK_OVERDUE) и блокировке (STATUS_CHANGED → BLOCKED) — прямые сигналы о проблемах потока. Своевременная реакция через UI сокращает время ожидания.
- **Traceability/audit**: Само уведомление — не аудит-событие. Состояние read/unread отслеживается в notification entity. Навигация по клику на уведомление ведёт к auditable entities (задачи, заказы).
- **Security/API boundary**: Все API-вызовы идут через JWT Bearer авторизацию. Пользователь видит только свои уведомления (recipientUserId = текущий userId из JWT). Фронтенд не делает security decisions — серверная фильтрация.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Пользователь видит badge с count непрочитанных в течение 30 секунд после появления нового уведомления.
- **SC-002**: Пользователь может открыть dropdown, увидеть последние уведомления и перейти на карточку объекта за 2 клика (колокольчик → уведомление).
- **SC-003**: Пользователь может пометить все уведомления прочитанными за 2 клика (колокольчик → «Отметить все»).
- **SC-004**: Страница уведомлений загружает первую страницу (20 элементов) без заметной задержки для пользователя.
- **SC-005**: Polling badge не создаёт заметного влияния на отзывчивость интерфейса.
- **SC-006**: Все типы уведомлений (TASK_ASSIGNED, STATUS_CHANGED, TASK_OVERDUE) визуально различимы по иконке.

## Assumptions

- Backend API уведомлений полностью реализован и работает (спеки 008 + 009).
- Для навигации по клику необходимо добавить поле `targetEntityId: UUID?` в backend NotificationResponse — минимальный backend change (FR-014). Без этого поля навигация невозможна, т.к. targetId содержит человекочитаемый номер (PT-000001), а маршруты требуют UUID.
- Polling (GET unread-count каждые 30 с) — достаточный механизм для near-realtime. WebSocket/SSE — out of scope (M7 Phase 1).
- Удаление уведомлений — out of scope. Только чтение и пометка прочитанными.
- Интернационализация: только русский язык (ru.ts). Все строки через i18n.
- Существующие UI-компоненты (shadcn-style из src/components/ui) используются без доработок.
