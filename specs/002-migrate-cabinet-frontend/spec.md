## User Scenarios & Testing *(mandatory)*

### User Story 1 - Запуск перенесенного кабинета в Docker (Priority: P1)

Разработчик запускает локальное окружение новой платформы и получает доступ к перенесенному
frontend-кабинету без запуска Frappe-версии. Кабинет открывается как самостоятельный интерфейс
новой платформы рядом с backend API.

**Why this priority**: это минимальный переносимый срез, который доказывает, что существующий
кабинет можно отделить от Frappe runtime и запускать в новой Spring/Kotlin платформе.

**Independent Test**: сценарий считается выполненным, если разработчик запускает локальное Docker
окружение и открывает frontend-адрес, где приложение загружается без белого экрана, ошибок загрузки
основных ассетов и зависимости от запущенного старого Frappe-приложения.

**Acceptance Scenarios**:

1. **Given** разработчик находится в корне нового проекта и имеет рабочую среду контейнеризации,
   **When** он запускает локальное окружение, **Then** backend API и frontend-кабинет стартуют в
   рамках одного локального Docker workflow.
2. **Given** локальное окружение запущено, **When** разработчик открывает frontend-адрес кабинета,
   **Then** браузер загружает перенесенное SPA-приложение и показывает экран входа.

---

### User Story 2 - Экран логина доступен без авторизации (Priority: P2)

Пользователь открывает кабинет и видит существующий экран логина, перенесенный из Frappe-версии.
Форма должна быть визуально доступна и интерактивна, но реальная авторизация пока не выполняется.

**Why this priority**: login screen является первым пользовательским состоянием кабинета и нужен для
проверки визуального переноса до реализации новой IAM/auth интеграции.

**Independent Test**: сценарий считается выполненным, если экран логина открывается напрямую и через
редирект с защищенного маршрута, поля доступны для ввода, а попытка входа не создает сессию и не
переводит пользователя в рабочий кабинет.

**Acceptance Scenarios**:

1. **Given** пользователь не авторизован, **When** он открывает кабинет, **Then** приложение
   показывает экран логина вместо рабочих страниц.
2. **Given** пользователь заполнил логин и пароль, **When** он отправляет форму, **Then** приложение
   сообщает, что авторизация в новой платформе пока не подключена, и остается на экране логина.
3. **Given** пользователь открыл защищенный маршрут кабинета напрямую, **When** приложение
   обрабатывает навигацию, **Then** оно переводит пользователя на login screen с сохранением
   безопасного пути возврата для будущей авторизации.

---

### User Story 3 - Сохранение переносимого UI-контента (Priority: P3)

Разработчик переносит существующие страницы, компоненты, стили, локализацию, assets и тестовые
артефакты кабинета так, чтобы они стали частью нового проекта и могли использоваться в следующих
feature-slice'ах.

**Why this priority**: после запуска login screen важно не потерять накопленную работу по layout,
темизации, компонентам, i18n и страницам заказов, даже если данные и авторизация будут
переинтегрированы позже.

**Independent Test**: сценарий считается выполненным, если структура frontend-кода присутствует в
новом проекте, собирается локально, а Frappe-specific интеграции изолированы так, чтобы не ломать
запуск login screen.

**Acceptance Scenarios**:

1. **Given** старый cabinet app содержит страницы, компоненты, стили, i18n и assets, **When**
   перенос завершен, **Then** эти артефакты доступны в новом frontend-контуре без обращения к старой
   директории Frappe-проекта.
2. **Given** часть старого frontend-кода обращалась к Frappe endpoints или socket runtime, **When**
   новый кабинет запускается без Frappe, **Then** такие обращения не блокируют отображение login
   screen и явно помечены как будущая backend-интеграция.

---

### Edge Cases

- Что происходит, если старый frontend содержит absolute paths или Frappe asset base path?
- Что происходит, если login page импортирует assets, которые не были перенесены?
- Как приложение ведет себя, если пользователь пытается открыть `/cabinet/orders` без авторизации?
- Как форма логина обрабатывает submit до появления новой auth-интеграции?
- Как локальный Docker workflow сообщает о неуспешной сборке frontend-контейнера?
- Как гарантируется, что новый frontend не обращается к старому Frappe runtime при первичном запуске?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Проект MUST включать перенесенный frontend cabinet app из старой Frappe-версии как
  часть нового репозитория, без runtime-зависимости от старой директории проекта.
- **FR-002**: Локальный Docker workflow MUST запускать backend API и frontend cabinet app одной
  согласованной командой из корня нового проекта.
- **FR-003**: Frontend cabinet app MUST открываться по документированному локальному адресу.
- **FR-004**: При открытии кабинета неавторизованный пользователь MUST видеть login screen.
- **FR-005**: Login screen MUST сохранять основные визуальные элементы старой версии: бренд-блок,
  форму ввода логина и пароля, состояние ошибки и базовую локализацию.
- **FR-006**: До реализации новой auth-интеграции отправка формы логина MUST NOT создавать
  пользовательскую сессию, MUST NOT выполнять успешный переход в рабочие разделы и MUST показывать
  понятное сообщение о временно недоступной авторизации.
- **FR-007**: Защищенные frontend-маршруты MUST перенаправлять неавторизованного пользователя на
  login screen.
- **FR-008**: Frontend startup MUST NOT требовать работающий старый Frappe backend для отображения
  login screen.
- **FR-009**: Перенесенный frontend MUST сохранять исходные страницы, компоненты, стили, i18n,
  assets и тестовые артефакты настолько полно, насколько это не блокирует запуск login screen.
- **FR-010**: Frappe-specific API, socket и boot-payload интеграции MUST быть изолированы,
  отключены или заменены безопасными временными заглушками для этого feature-slice.
- **FR-011**: Docker workflow MUST предоставлять способ проверить готовность frontend-сервиса и
  посмотреть его логи.
- **FR-012**: Документация MUST описывать frontend URL, запуск, остановку, логи и текущие
  ограничения авторизации.
- **FR-013**: Backend API MUST оставаться API-only и не должен снова показывать browser login page
  или Basic Auth popup из-за добавления frontend.
- **FR-014**: Перенос MUST NOT реализовывать новую backend-auth схему, управление пользователями или
  выдачу токенов; это отдельный future feature-slice.

### Key Entities

- **Frontend Cabinet App**: перенесенное SPA-приложение кабинета с routes, layout, pages,
  components, styles, i18n и assets.
- **Login Screen**: публичное состояние frontend-приложения для неавторизованного пользователя.
- **Authentication Placeholder**: временное поведение формы входа, которое объясняет, что новая
  авторизация пока не подключена.
- **Docker Frontend Service**: локальный runtime-сервис, который собирает или обслуживает frontend в
  Docker workflow.
- **Legacy Integration Boundary**: места старого frontend-кода, завязанные на Frappe endpoints,
  socket runtime или boot-payload, которые должны быть отделены от первого запуска.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Новый разработчик может запустить backend и frontend локально одной командой за 15
  минут или меньше после установки предварительных требований.
- **SC-002**: Frontend login screen становится доступен в браузере за 2 минуты или меньше после
  завершения локального запуска.
- **SC-003**: 100% обязательных assets login screen загружаются без ошибок в браузере.
- **SC-004**: Попытка входа до реализации auth-интеграции в 100% случаев остается на login screen и
  показывает понятное сообщение вместо успешной авторизации.
- **SC-005**: Открытие защищенного frontend-маршрута неавторизованным пользователем в 100% случаев
  приводит к login screen.
- **SC-006**: Перенесенный frontend запускается без обращения к старому Frappe runtime для первого
  отображения login screen.

## Assumptions

- Перенос выполняется в новый проект `ctfind-production-control-contlin`, а старая Frappe-директория
  используется только как source reference.
- Первая цель переноса — работающий frontend runtime и login screen, а не полная интеграция с новой
  backend domain model.
- Авторизация, роли, реальные session/token flows и backend endpoints будут спроектированы отдельным
  feature-slice.
- Рабочие страницы кабинета могут быть перенесены как код и маршруты, но их данные могут оставаться
  недоступными или заглушенными до появления новых API.
- Docker workflow для backend из `001-local-docker-startup` считается базой для добавления frontend
  service.
