# Feature Specification: Журнал действий администратора (Phase 1 §8 #10)

**Feature Branch**: `007-audit-log-viewer`
**Created**: 2026-04-28
**Status**: Pending verification
**Input**: User description: "Brief: Журнал действий администратора (Phase 1 §8 #10) — единая страница для администратора, показывающая cross-entity ленту аудита (auth + orders + production tasks) с фильтрами по дате/категории/исполнителю/поиску, пагинацией и переходом на карточку объекта; читает существующие три аудит-таблицы, не вводит новых аудит-источников."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Администратор видит сводный журнал действий за неделю (Priority: P1)

Администратор открывает «Журнал действий» в кабинете и сразу видит ленту последних событий системы за последние 7 дней: входы/выходы пользователей, действия по заказам, изменения по производственным задачам. Каждая строка показывает «когда / кто / категория / тип события / краткое описание / над каким объектом». По клику на строку с привязкой к заказу или к производственной задаче открывается карточка соответствующего объекта.

**Why this priority**: Это последний оставшийся критерий завершения Phase 1 (§8 #10). Без сводного журнала администратор может изучить аудит только заходя в каждый отдельный заказ / задачу — это не масштабируется и не закрывает требование «просматривать журнал действий» по системе в целом.

**Independent Test**: Под пользователем `admin / admin` после логина в кабинете доступна навигационная точка «Журнал действий»; при её открытии отображается лента из 3 категорий событий за последние 7 дней, отсортированная по времени по убыванию. Клик по событию из категории `ORDER` открывает карточку заказа; клик по событию из категории `PRODUCTION_TASK` открывает карточку задачи; клик по событию из категории `AUTH` ничего не делает (или явно неактивен). Без бэкенд-данных (свежая БД) лента может быть пустой — в этом случае показывается пустое состояние.

**Acceptance Scenarios**:

1. **Given** ADMIN залогинен в кабинете, в системе есть как минимум по одному событию каждой из трёх категорий за последние 7 дней, **When** ADMIN открывает «Журнал действий», **Then** видит таблицу с этими событиями, отсортированную по времени по убыванию, и видит для каждой строки время, категорию, тип события, имя инициатора и краткое описание.
2. **Given** на странице открыта лента, **When** ADMIN кликает по строке с категорией `ORDER`, **Then** в кабинете открывается карточка заказа (`orders.detail`) с тем же `targetId`.
3. **Given** на странице открыта лента, **When** ADMIN кликает по строке с категорией `PRODUCTION_TASK`, **Then** открывается карточка задачи (`production-tasks.detail`).
4. **Given** на странице открыта лента, **When** ADMIN кликает по строке категории `AUTH`, **Then** ничего не происходит (ссылка отсутствует) — это ожидаемое поведение, фиксируется в acceptance.
5. **Given** в системе нет ни одного аудит-события за последние 7 дней, **When** ADMIN открывает «Журнал действий», **Then** видит пустое состояние с пояснением «Событий за выбранный период нет».

### User Story 2 — Администратор сужает выборку фильтрами (Priority: P2)

Администратор открыл журнал и хочет понять, что делал конкретный исполнитель за вчерашний день — или только увидеть события входа в систему за неделю. Над лентой есть панель фильтров: диапазон дат, категории (multi-select), исполнитель (actor user picker), поле текстового поиска. Любая комбинация фильтров мгновенно применяется к ленте; кнопка «Сбросить фильтры» возвращает дефолтное состояние (последние 7 дней, без других ограничений). Между страницами навигация постраничная (по 50 записей на страницу), с индикатором общего количества и текущей страницы.

**Why this priority**: Без фильтров на больших объёмах журнала ADMIN утонет в ленте; цель — расследование инцидентов и точечный аудит, а не лента ради ленты. Это **расширение** P1, без P1 не имеет смысла.

**Independent Test**: На фоне готовой ленты из P1 — изменение каждого фильтра по очереди (диапазон, категория, исполнитель, поиск) меняет количество строк; «Сбросить» возвращает к дефолту; пагинатор работает между страницами.

**Acceptance Scenarios**:

1. **Given** ADMIN видит ленту из 200 событий за последние 7 дней, **When** ADMIN выбирает в категориях только `AUTH`, **Then** в ленте остаются только события входа/выхода/seed; счётчик `всего` обновляется.
2. **Given** ADMIN видит ленту, **When** ADMIN выбирает в фильтре «Исполнитель» конкретного пользователя, **Then** в ленте остаются только события, инициированные этим пользователем.
3. **Given** ADMIN видит ленту, **When** ADMIN сужает диапазон дат до «вчера», **Then** в ленте остаются события только за этот день.
4. **Given** ADMIN видит ленту с 250 событиями, размер страницы 50, **When** ADMIN переключается на 2-ю страницу, **Then** видит события 51–100, кнопка «Назад» становится активной.
5. **Given** ADMIN применил несколько фильтров, **When** ADMIN жмёт «Сбросить», **Then** все фильтры возвращаются к дефолтным значениям (последние 7 дней, без категорий/исполнителя/поиска), лента перезагружается.
6. **Given** комбинация фильтров не даёт ни одной строки, **When** результат пустой, **Then** ADMIN видит пустое состояние с подсказкой «Под фильтры ничего не подходит — измените или сбросьте».

### User Story 3 — Журнал недоступен ролям без прав (Priority: P3)

Пользователь с ролью `ORDER_MANAGER`, `PRODUCTION_SUPERVISOR` или `PRODUCTION_EXECUTOR` не должен ни видеть пункт «Журнал действий» в навигации, ни попасть на страницу прямым URL: при попытке прямого перехода система должна вернуть состояние «Нет доступа» (403) и не раскрыть никаких данных журнала.

**Why this priority**: Журнал содержит сводную информацию по всей системе, включая аутентификационные события; раскрытие неуполномоченным ролям нарушает базовые ожидания по разделению полномочий. Это **дополнение к гарантиям** P1 (которые сами по себе работают только под ADMIN). Без P3 нарушается security boundary.

**Independent Test**: Логин под `production.executor / executor`, под `production.supervisor / supervisor` и под `order.manager / manager` — в навигации нет «Журнала действий»; прямой URL `/cabinet/audit` не отдаёт данные (либо редиректит на запретный экран, либо показывает forbidden state, либо страница попросту не зарегистрирована в роутинге для этих ролей — конкретный механизм определяется в плане).

**Acceptance Scenarios**:

1. **Given** пользователь с ролью `PRODUCTION_EXECUTOR` залогинен, **When** он смотрит главную навигацию кабинета, **Then** пункта «Журнал действий» в меню нет.
2. **Given** пользователь с ролью `ORDER_MANAGER` залогинен, **When** он переходит по прямому URL `/cabinet/audit`, **Then** видит экран «Нет доступа» (или эквивалентный forbidden state) и не получает данных журнала.
3. **Given** пользователь без аутентификации (без активного JWT), **When** он обращается к источнику данных журнала напрямую, **Then** ответ помечен как «не аутентифицирован» (401), а не как «нет такого ресурса» или «доступ запрещён».

### Edge Cases

- Очень большое количество событий за выбранный период (десятки тысяч): пагинация показывает только текущую страницу, общий `totalItems` отражает реальное количество, лента не пытается загрузить всё разом.
- В строке журнала указан `targetId`, но соответствующий объект уже удалён или недоступен текущему пользователю: клик не должен привести к необработанной ошибке. Допустимое поведение в Phase 1: переход на карточку, и карточка сама показывает 404 / forbidden — это воспринимаемое поведение, журнал не обязан валидировать существование объекта на клиенте.
- Событие категории `AUTH`, у которого `actorUserId` относится к удалённому пользователю: в строке журнала отображается `actorDisplayName` снепшотом из самого события (если в источнике сохранён); если нет — показывается логин из события или плейсхолдер «Удалённый пользователь».
- Сетевая ошибка при загрузке страницы журнала: ADMIN видит баннер ошибки и кнопку «Обновить», но не пустую таблицу без объяснений.
- Включён фильтр «диапазон дат», у которого `from > to`: UI должен либо не позволять выбрать такую комбинацию, либо считать её эквивалентом «нет совпадений» (пустая лента + подсказка), но не падать.
- Фильтр «исполнитель» выбирает пользователя, у которого нет ни одного события в системе: лента пустая, плюс плейсхолдер.
- ADMIN листает на N-ю страницу, потом меняет фильтр: страница сбрасывается в 1-ю (чтобы не оказаться в пустоте «11-я страница, но новый фильтр даёт всего 3 строки»).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять одну точку входа в кабинете — страницу «Журнал действий» по маршруту `/cabinet/audit`, доступную в меню навигации.
- **FR-002**: Доступ к странице и к её данным MUST быть открыт только пользователям с ролью `ADMIN`. Пользователи с ролями `ORDER_MANAGER`, `PRODUCTION_SUPERVISOR`, `PRODUCTION_EXECUTOR`, а также неаутентифицированные пользователи MUST получать состояние «Нет доступа» (для роли — 403, без аутентификации — 401) и не получать никаких данных журнала.
- **FR-003**: Пункт «Журнал действий» в навигации MUST быть скрыт от пользователей без роли `ADMIN`.
- **FR-004**: Журнал MUST быть единой лентой событий, объединяющей три источника аудита, существующие в системе: события аутентификации, аудит-события по заказам, аудит-события по производственным задачам. Никакие новые источники аудита данной фичей не вводятся.
- **FR-005**: Каждая строка журнала MUST содержать: момент времени события, категорию (`AUTH` / `ORDER` / `PRODUCTION_TASK`), тип события, отображаемое имя инициатора, краткое читаемое описание события, тип целевого объекта (если применимо для категории) и идентификатор целевого объекта (если применимо).
- **FR-006**: Строка журнала с категорией `ORDER` MUST содержать переход на карточку заказа (`orders.detail`) с тем же `targetId`. Строка с категорией `PRODUCTION_TASK` MUST содержать переход на карточку производственной задачи (`production-tasks.detail`). Строка с категорией `AUTH` НЕ содержит ссылки.
- **FR-007**: По умолчанию журнал MUST показывать события за последние 7 дней относительно момента открытия страницы.
- **FR-008**: Лента журнала MUST быть отсортирована по времени события по убыванию (самые свежие сверху).
- **FR-009**: Журнал MUST поддерживать страничную пагинацию по 50 событий на страницу с индикаторами «текущая страница / всего страниц» и «всего событий», и кнопками «вперёд / назад / на конкретную страницу».
- **FR-010**: Журнал MUST поддерживать фильтр по диапазону дат (`from`, `to`); по умолчанию — последние 7 дней. UI MUST не позволять выбрать `from > to` либо в этом случае показывать пустой результат с подсказкой.
- **FR-011**: Журнал MUST поддерживать фильтр по категории — multi-select из `AUTH`, `ORDER`, `PRODUCTION_TASK`. По умолчанию все три категории включены.
- **FR-012**: Журнал MUST поддерживать фильтр по конкретному инициатору события (выбор пользователя из системы).
- **FR-013**: Журнал MUST поддерживать текстовый поиск режимом substring ILIKE по трём полям: краткое описание события (summary), идентификатор целевого объекта (targetId) и логин актора (actorLogin). Полнотекстовый поиск (tsvector/tsquery) не требуется в Phase 1.
- **FR-014**: Любая смена набора фильтров MUST автоматически сбрасывать пагинацию на 1-ю страницу, чтобы пользователь не остался на «несуществующей» странице.
- **FR-015**: На странице MUST быть кнопка «Сбросить фильтры», возвращающая фильтры к дефолтному состоянию (последние 7 дней, все категории включены, исполнитель не выбран, поиск пуст).
- **FR-016**: На странице MUST быть кнопка «Обновить», перечитывающая ленту с актуальными фильтрами; перечитывание MUST также происходить при первом монтировании страницы.
- **FR-017**: Когда журнал не может быть загружен из-за сетевой ошибки или ответа об ошибке от источника данных, страница MUST показать дискретное состояние ошибки с пояснением и кнопкой «Обновить», не показывая стейл-данные молча.
- **FR-018**: Когда сервер вернул признак «нет доступа», страница MUST показать дискретное состояние «Нет доступа» с пояснением и без раскрытия каких-либо данных журнала.
- **FR-019**: Страница MUST корректно отображаться на планшете 10–12" в landscape (адаптация под §8 общих требований Phase 1): фильтр-панель сворачивается на узких экранах, таблица событий допускает горизонтальный скролл.
- **FR-020**: Журнал MUST соблюдать неизменяемость источников: эта фича только читает аудит-данные, не пишет новых событий, не меняет существующие, не удаляет.
- **FR-021**: Каждая строка журнала содержит только сводное поле «описание» (summary). Раскрытие сырого метаданного события (JSON / payload) и drill-down с форматтером по типу события НЕ входят в Phase 1. Детальная информация доступна через переход на карточку объекта (FR-006).
- **FR-022**: Журнал MUST быть представлен единой таблицей с общими колонками для всех трёх категорий событий. Фильтрация по категории осуществляется через multi-select (FR-011), без разделения на вкладки.

### Key Entities *(view-model)*

- **Audit log row**: одна запись объединённой ленты. Атрибуты: момент события, категория (одна из трёх), тип события, инициатор события (его отображаемое имя), краткое описание события, тип целевого объекта (если применимо), идентификатор целевого объекта (если применимо), признак «есть переход на карточку» (для `ORDER` и `PRODUCTION_TASK`).
- **Audit log filter set**: текущая комбинация фильтров. Атрибуты: диапазон дат (`from`, `to`), включённые категории (подмножество `{AUTH, ORDER, PRODUCTION_TASK}`), идентификатор выбранного инициатора (опционально), строка поиска (опционально), текущая страница, размер страницы.
- **Audit log page**: одна страница ленты, возвращаемая источником данных. Атрибуты: список строк, общее количество событий под текущим набором фильтров, общее количество страниц, индекс текущей страницы, размер страницы.

### Constitution Alignment *(mandatory)*

- **ERP/domain fit**: фича закрывает критерий «просматривать журнал действий» из §8 Phase 1, давая администратору единую точку наблюдения за операторскими действиями в трёх ключевых модулях системы (auth, orders, production tasks). Без неё журнал размазан по детальным карточкам — невозможно расследовать инциденты, у которых пересекаются несколько объектов.
- **TOC readiness**: фича не вводит новых фактов о потоке (статусы, времена ожидания, throughput) — она только публикует уже существующие, написанные в момент действий. Существующая инфраструктура аудит-событий по производственным задачам и заказам не меняется; этот журнал — read-only-витрина над ними.
- **Traceability/audit**: эта фича **и есть** реализация traceability/audit на уровне UI. Источники аудит-событий (`auth_audit_event`, `order_audit_event`, `production_task_audit_event`) уже пишутся транзакционно вместе с действиями (см. возможности, реализованные в Phase 1 ранее); фича их не трогает, только читает. Никаких новых auditable mutations не вводится.
- **Security/API boundary**: журнал — admin-only по дизайну. Серверный слой данных MUST отвечать `403` любому пользователю без роли `ADMIN` и `401` неаутентифицированному пользователю. Никаких клиентских ролей-проверок (v-if по роли) на странице — гейт делается только на сервере и в маршрутизации (ролевой guard на маршруте `/cabinet/audit`). Фича не вводит новых публичных API-точек для аутентификации/авторизации, не трогает JWT/RBAC-механику, не меняет ролевую модель.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ADMIN, открывший «Журнал действий», MUST увидеть нужное событие из последних 7 дней (если оно произошло) не более чем за 30 секунд от открытия страницы — за счёт сортировки по времени по убыванию + пагинации по 50.
- **SC-002**: При расследовании инцидента ADMIN MUST уметь сузить ленту до релевантной выборки за **не более 4 действий** (выбрать категорию + выбрать исполнителя + задать диапазон дат + при необходимости поиск).
- **SC-003**: Переход с записи журнала на карточку соответствующего объекта (`ORDER` или `PRODUCTION_TASK`) MUST быть выполним **в один клик**.
- **SC-004**: Доля попыток просмотра журнала пользователями без роли `ADMIN`, в которых журнал хоть как-то раскрывает данные, MUST быть ровно **0%** — это закрывается контрактом «403 на не-ADMIN, навигация скрыта».
- **SC-005**: Страница MUST штатно отображать ленту длиной не менее **10 000 событий** в БД для выбранного интервала, без перегрузки браузера — это обеспечивается пагинацией по 50 и серверной выборкой по странице.
- **SC-006**: По завершении фичи **§8 пункт #10** Phase 1 (`docs/PHASE_01.md`) MUST быть помечен как закрытый, а Phase 1 MUST быть полностью завершена по чек-листу §8.

## Clarifications

### Session 2026-04-28

- Q: Единая таблица под мульти-фильтр по категории, или 3 вкладки Auth/Orders/Production Tasks с разными колонками? → A: Единая таблица с общими колонками + multi-select фильтр по категории.
- Q: Должен ли текстовый поиск покрывать логин актора в auth-событиях с substring-матчем, или достаточно structured ILIKE по описанию + targetId? → A: Substring ILIKE по summary + targetId + логину актора.
- Q: Нужен ли в Phase 1 разворачиваемый drill-down с metadata-форматтером по типу события, или достаточно summary-поля на строку? → A: Только summary на строку, без drill-down. Детали — через переход на карточку объекта.

## Assumptions

- ADMIN — единственный пользователь P1. Расширение доступа на `ORDER_MANAGER` или `PRODUCTION_SUPERVISOR` (например, ограниченный вид только их сущностей) **не входит** в Phase 1; обсуждается отдельно как Phase 2 или последующее уточнение, но в данной спеке **не реализуется** (см. также вопрос Q4 в брифе).
- Источники аудит-данных (три аудит-таблицы и более гранулярные таблицы изменений) уже пишутся транзакционно с действиями в текущем коде Phase 1; данная фича только читает существующие данные. Никаких новых записывающих кодовых путей не добавляется.
- Дефолтный диапазон «последние 7 дней» считается разумным под рабочую нагрузку администратора в Phase 1; при необходимости пользователь сужает или расширяет диапазон вручную через фильтр.
- Сортировка только по времени события по убыванию. Кликабельная смена сортировки по другим колонкам (исполнитель, категория) **не входит** в Phase 1.
- Размер страницы фиксированный — 50 строк. Изменяемый размер страницы и infinite scroll **не входят** в Phase 1.
- UI русскоязычный, как и остальной кабинет.
- Экспорт журнала (CSV / XLSX), retention / автоочистка старых событий, подпись / целостность аудита (HMAC-цепочка), realtime-обновление журнала, drill-down фильтр «всё, что делал actor X» как отдельная UX-фича — все **out of scope** Phase 1, явно фиксируются как Phase 2+.
- Существование объекта, на который ссылается строка журнала (заказ / производственная задача), не валидируется на клиенте до клика; на момент клика карточка объекта сама обрабатывает 404 / forbidden — это допустимое граничное поведение Phase 1.
- Адаптация под планшет 10–12" landscape — обязательная (§8 общих требований Phase 1); адаптация под телефон **не входит** в Phase 1.
