# Feature Specification: Триггеры генерации уведомлений

**Feature Branch**: `009-notification-triggers`  
**Created**: 2026-04-29  
**Status**: Accepted  
**Dependency**: 008-notifications-infrastructure (domain, persistence, REST API)

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Уведомление о назначении задачи (Priority: P1)

Мастер (PRODUCTION_SUPERVISOR) назначает производственную задачу на исполнителя. Исполнитель (PRODUCTION_EXECUTOR) получает уведомление «Вам назначена задача PT-001» в своей ленте уведомлений. Уведомление содержит ссылку на задачу (targetType=PRODUCTION_TASK, targetId=taskNumber).

**Why this priority**: Назначение задачи — основной триггер рабочего процесса. Без уведомления исполнитель может не узнать о новой задаче вовремя, что блокирует выполнение работ.

**Independent Test**: Можно проверить изолированно: назначить задачу на исполнителя, затем запросить GET /api/notifications от имени исполнителя и убедиться, что уведомление типа TASK_ASSIGNED появилось в ленте.

**Acceptance Scenarios**:

1. **Given** задача PT-001 без исполнителя, **When** мастер назначает исполнителя Иванова, **Then** Иванов получает уведомление с type=TASK_ASSIGNED, title содержит «PT-001», targetType=PRODUCTION_TASK, targetId=«PT-001»
2. **Given** задача PT-001 назначена на Иванова, **When** мастер переназначает задачу на Петрова, **Then** Петров получает уведомление TASK_ASSIGNED (новый исполнитель)
3. **Given** мастер назначает задачу, **When** исполнитель запрашивает список уведомлений, **Then** уведомление отображается с read=false
4. **Given** задача PT-001 назначена на Иванова, **When** мастер повторно сохраняет задачу без смены исполнителя, **Then** повторное уведомление НЕ создаётся

---

### User Story 2 — Уведомление о смене статуса задачи (Priority: P1)

Исполнитель меняет статус задачи (например, начинает работу или завершает задачу). Мастер, создавший задачу, получает уведомление «Задача PT-001: статус изменён на IN_PROGRESS». Если пользователь меняет статус собственной задачи и он же является создателем — уведомление не создаётся (нет смысла уведомлять себя).

**Why this priority**: Мастер должен отслеживать прогресс задач без необходимости постоянно обновлять доску. Оповещения о смене статуса — ключевой инструмент пассивного мониторинга.

**Independent Test**: Можно проверить изолированно: сменить статус задачи от имени исполнителя, затем запросить уведомления от имени мастера-создателя.

**Acceptance Scenarios**:

1. **Given** задача PT-001 создана мастером Сидоровым и назначена на Иванова, **When** Иванов меняет статус на IN_PROGRESS, **Then** Сидоров получает уведомление с type=STATUS_CHANGED, title содержит «PT-001» и «IN_PROGRESS»
2. **Given** задача PT-001 создана Сидоровым, **When** Сидоров сам меняет статус задачи на COMPLETED, **Then** уведомление НЕ создаётся (sender === recipient)
3. **Given** задача PT-001 переходит в статус BLOCKED, **When** исполнитель блокирует задачу, **Then** создатель получает уведомление STATUS_CHANGED с новым статусом BLOCKED

---

### User Story 3 — Уведомление о просрочке задачи (Priority: P2)

Система периодически проверяет задачи с истёкшей плановой датой завершения (plannedFinishDate < сегодня), которые ещё не завершены. Для каждой такой задачи отправляется уведомление исполнителю и мастеру-создателю. Повторные уведомления не создаются — для каждой задачи TASK_OVERDUE отправляется один раз.

**Why this priority**: Просрочки — важный сигнал, но требуют фонового процесса (@Scheduled). Менее критично чем мгновенные триггеры (US1, US2), потому что просроченные задачи уже видны на доске.

**Independent Test**: Можно проверить изолированно: создать задачу с plannedFinishDate в прошлом, запустить scheduled job, затем проверить что уведомления TASK_OVERDUE появились у исполнителя и создателя.

**Acceptance Scenarios**:

1. **Given** задача PT-001 с plannedFinishDate=вчера и статусом IN_PROGRESS, **When** запускается проверка просрочек, **Then** исполнитель и создатель получают уведомления type=TASK_OVERDUE
2. **Given** задача PT-001 уже имеет уведомление TASK_OVERDUE, **When** проверка просрочек запускается повторно, **Then** повторное уведомление НЕ создаётся (дедупликация)
3. **Given** задача PT-001 с plannedFinishDate=вчера и статусом COMPLETED, **When** запускается проверка, **Then** уведомление НЕ создаётся (задача завершена)
4. **Given** задача без назначенного исполнителя с просроченной датой, **When** проверка запускается, **Then** уведомление получает только создатель задачи

---

### Edge Cases

- Задача назначена, но исполнитель удалён из системы → уведомление не создаётся (FK constraint, пользователь должен существовать)
- Мастер назначает задачу на самого себя → уведомление создаётся (назначение отличается от смены статуса, мастер в роли исполнителя должен видеть назначение)
- Одновременная смена статуса и переназначение → оба уведомления создаются (разные триггеры, разные получатели)
- Задача без plannedFinishDate → не считается просроченной (null-дата не сравнивается)
- Ошибка при создании уведомления → основная операция (назначение, смена статуса) НЕ прерывается; ошибка логируется
- Scheduled job: при большом количестве просроченных задач обработка идёт последовательно, нет batch-limit на данном этапе

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST создавать уведомление TASK_ASSIGNED при назначении исполнителя на задачу (executorUserId меняется с null на значение или с одного значения на другое)
- **FR-002**: Система MUST создавать уведомление STATUS_CHANGED при изменении статуса задачи, адресованное создателю задачи (createdByUserId)
- **FR-003**: Система MUST NOT создавать уведомление STATUS_CHANGED если инициатор смены статуса является создателем задачи (self-notification suppression)
- **FR-004**: Система MUST периодически проверять задачи с просроченной плановой датой завершения (plannedFinishDate < текущая дата) и статусом отличным от COMPLETED
- **FR-005**: Система MUST создавать уведомление TASK_OVERDUE для исполнителя и создателя просроченной задачи
- **FR-006**: Система MUST NOT создавать повторное уведомление TASK_OVERDUE для задачи, по которой уведомление уже было отправлено
- **FR-007**: Система MUST NOT прерывать основную бизнес-операцию при ошибке создания уведомления (fire-and-forget с логированием)
- **FR-008**: Система MUST NOT создавать уведомление TASK_ASSIGNED при сохранении задачи без фактической смены исполнителя
- **FR-009**: Система MUST использовать кросс-модульный порт (NotificationCreatePort) для создания уведомлений — прямой импорт внутренних классов модуля notifications запрещён
- **FR-010**: Триггеры назначения и смены статуса MUST срабатывать синхронно в рамках use-case модуля production

### Key Entities

- **Notification** (из спеки 008): уведомление с полями type, title, body, targetType, targetId, recipientUserId, read, createdAt
- **ProductionTask**: производственная задача с полями executorUserId (назначенный исполнитель), createdByUserId (создатель/мастер), status, taskNumber, plannedFinishDate
- **NotificationType**: перечисление типов — TASK_ASSIGNED, STATUS_CHANGED, TASK_OVERDUE (VARCHAR storage, расширение без миграции)

### Constitution Alignment *(mandatory)*

- **ERP/domain fit**: Укрепляет рабочий процесс производственных задач — ключевой операционной сущности. Уведомления связывают назначение, статусы и дедлайны с конкретными людьми, превращая пассивный контроль в активный.
- **TOC readiness**: TASK_OVERDUE напрямую связан с due-date pressure (plannedFinishDate). Факт отправки overdue-уведомления фиксирует момент осознания задержки, что ценно для анализа буферов. STATUS_CHANGED уведомления трассируют flow history через модуль уведомлений.
- **Traceability/audit**: Каждое уведомление сохраняется в таблицу notification с createdAt, type, targetId — это аудируемая запись о событии. Audit log модуля production фиксирует сами операции; уведомления дополняют их информацией о том, кто был проинформирован.
- **Security/API boundary**: Триггеры работают только внутри backend — нет новых API endpoints. Существующие REST endpoints из спеки 008 остаются единственной точкой доступа. Кросс-модульный доступ через порт (interface), не через прямой импорт.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: При назначении задачи на исполнителя уведомление появляется в ленте получателя в течение 1 секунды (синхронный триггер)
- **SC-002**: При смене статуса задачи уведомление появляется в ленте создателя задачи в течение 1 секунды (синхронный триггер)
- **SC-003**: Просроченные задачи обнаруживаются и уведомления генерируются в течение 15 минут (интервал scheduled job)
- **SC-004**: Ни один тип уведомления не дублируется: TASK_OVERDUE генерируется ровно один раз на задачу, TASK_ASSIGNED — только при фактической смене исполнителя
- **SC-005**: Ошибка генерации уведомления ни в одном сценарии не приводит к сбою основной операции (назначение, смена статуса)

## Assumptions

- Спека 008-notifications-infrastructure полностью реализована: домен Notification, NotificationCreatePort, CreateNotificationUseCase, persistence, REST API
- Модуль production содержит use-case'ы AssignProductionTaskUseCase и ChangeProductionTaskStatusUseCase с доступом к текущим и предыдущим значениям
- Поле plannedFinishDate (LocalDate?) используется как дедлайн для определения просрочки
- Статус COMPLETED является единственным финальным статусом (BLOCKED — временный, задача может быть разблокирована)
- Приложение работает в одном экземпляре — distributed lock для scheduled job не требуется
- Часовой пояс сравнения дедлайна — серверный (UTC по умолчанию)
- Дедупликация TASK_OVERDUE реализуется через проверку существующих уведомлений (query по type + targetId + recipientUserId)
