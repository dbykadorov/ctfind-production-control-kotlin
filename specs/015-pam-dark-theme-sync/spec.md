# Feature Specification: Синхронизация тёмной темы кабинета с эталоном PAM UC

**Feature Branch**: `015-pam-dark-theme-sync`  
**Created**: 2026-04-30  
**Status**: Pending verification  
**Input**: User description: синхронизация тёмной темы Production Control Cabinet с визуальной и семантической моделью PAM User Console (палитра, оверлей, состояния UI); сохранение amber-бренда и floating-card layout; отступления от эталона документируются.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Единообразная тёмная тема с корпоративным эталоном (Priority: P1)

Оператор работает в кабинете в тёмном режиме и воспринимает фон приложения, боковую навигацию, верхнюю панель, таблицы, формы и модальные окна как визуально согласованные с утверждённым эталоном другого консольного продукта организации (PAM User Console): те же «уровни» поверхностей, читаемость текста, контуры полей и затемнение фона при диалогах.

**Why this priority**: Снижает когнитивную нагрузку при переключении между продуктами и поддерживает брендированную дисциплину интерфейса.

**Independent Test**: Набор ключевых экранов кабинета в тёмном режиме проверяется по утверждённой матрице соответствия эталону; расхождения либо устранены, либо явно перечислены как принятые отступления.

**Acceptance Scenarios**:

1. **Given** пользователь включил тёмную тему, **When** он открывает обзорный экран и основные рабочие разделы из согласованного списка QA, **Then** цвета фона, текста и разделителей соответствуют матрице соответствия эталону в пределах задокументированных допусков.
2. **Given** пользователь в тёмной теме открывает модальное окно, **When** диалог показан, **Then** затемнение фона по смыслу совпадает с политикой оверлея эталона (допуски по интенсивности задокументированы).
3. **Given** пользователь взаимодействует с отключёнными элементами и наведением на интерактивные элементы в типовых формах и таблицах, **Then** визуальные состояния согласованы с правилами эталона для disabled/hover в пределах принятой модели компонентов кабинета.

---

### User Story 2 — Сохранение узнаваемости продукта CTfind (Priority: P1)

Оператор по-прежнему узнаёт кабинет CTfind: акцентный цвет бренда (amber) и модель «плавающая карточка» контента остаются непротиворечивыми с текущей продуктовой линией; любые отличия от эталона PAM по акценту или форме контентной области явно описаны в матрице отступлений.

**Why this priority**: Без этого синхронизация палитры может расцениться как потеря идентичности продукта.

**Independent Test**: Зафиксированный чеклист бренда (акцент, карточка контента) проходит ревью владельца продукта или дизайна.

**Acceptance Scenarios**:

1. **Given** включена тёмная тема, **When** пользователь просматривает ключевые экраны, **Then** акцентные элементы остаются в домене утверждённого amber-бренда CTfind без замены на палитру акцента эталона, если иное не зафиксировано как отступление.
2. **Given** включена тёмная тема, **When** пользователь видит основную область контента внутри карточки приложения, **Then** поведение «elevated island» сохраняется (радиус/тень/отделение от chrome), а тон поверхности карточки определяется матрицей соответствия PAM.

---

### User Story 3 — Переключение темы без отвлекающих артефактов (Priority: P2)

При смене светлой и тёмной темы пользователь не видит длительного «мигания» или промежуточных неверных цветов из-за анимаций перехода на уровне цвета.

**Why this priority**: Улучшает восприятие качества интерфейса и совпадает с паттерном эталона (подавление transition при смене темы).

**Independent Test**: Ручная смена темы несколько раз подряд не демонстрирует заметных артефактов (согласно критериям QA).

**Acceptance Scenarios**:

1. **Given** пользователь переключает тему, **When** переключение завершено, **Then** не наблюдается устойчивого визуального артефакта из-за одновременной анимации цвета фона, текста и границ в течение переключения (критерии фиксируются в чеклисте QA).

---

### Edge Cases

- Пользователь с включённым режимом уменьшенного движения в ОС: переключение темы не должно ухудшать доступность.
- Сочетание тёмной темы с пользовательскими градиентными пресетами боковой панели: не должно появляться неконтрастных сочетаний; регрессии по пресетам недопустимы без явного отступления в матрице.
- Страница входа (если общая с кабинетом по темам): согласованность тёмного режима с основным приложением или явное исключение в матрице.
- Долгие сессии с автосохранением темы: повторная загрузка сохраняет ожидаемый внешний вид.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Продукт MUST предоставлять утверждённую матрицу соответствия между семантическими ролями цветов эталонной тёмной темы и ролями цветов кабинета CTfind (фон приложения, поверхности, текст сильный/обычный/приглушённый, границы, оверлей, ключевые статусы), с явным перечнем допустимых отклонений и обоснованием.
- **FR-002**: Тёмная тема кабинета MUST сохранять доминирующий акцент бренда CTfind (amber), если в матрице отступлений не зафиксировано иное по согласованию с владельцем продукта/дизайна.
- **FR-003**: В тёмной теме MUST сохраняться модель основной контентной области как «elevated island» на фоне chrome; конкретный тон поверхности карточки (light или dark) определяется матрицей соответствия и проверкой контраста.
- **FR-004**: Продукт MUST обеспечивать читаемость основного текста и интерактивных элементов в тёмной теме не ниже уровня WCAG 2.1 AA для типовых комбинаций «текст на фоне» и «состояние фокуса/контраст границ», задокументированных в матрице проверки.
- **FR-005**: Продукт MUST preserve отсутствие регрессий для пользовательских пресетов оформления боковой панели: каждый поддерживаемый пресет остаётся пригодным для использования в тёмной теме либо явно помечен ограничением в матрице.
- **FR-006**: Продукт MUST сохранять узнаваемость семантики статусных цветов (новый, в работе, готов, отгружен и др.) в тёмной теме; любые изменения оттенков фиксируются в матрице с проверкой различимости.
- **FR-007**: Поведение модальных окон в тёмной теме MUST соответствовать политике затемнения фона эталона по смыслу (непрозрачность/контраст с окружением), с задокументированными допусками.
- **FR-008**: Переключение между светлой и тёмной темой MUST выполняться без устойчивых визуальных артефактов, вызванных одновременной анимацией цветовых свойств; при необходимости применяется паттерн кратковременного подавления transitions на время смены темы (по аналогии с эталоном).
- **FR-009**: Область работ MUST исключать полный визуальный порт всех экранов PAM UC, замену UI-стека на PrimeVue и синхронизацию светлой темы с PAM, если это не оформлено отдельным запросом.
- **FR-010**: Должен существовать согласованный с заинтересованными сторонами минимальный набор экранов кабинета для обязательного QA sign-off после изменений тёмной темы.

### Key Entities

- **Тема интерфейса (светлая/тёмная)**: пользовательская настройка отображения, влияющая на палитру кабинета.
- **Матрица соответствия эталону**: документ сопоставления ролей цветов и состояний UI между эталоном и кабинетом, включая допуски и отступления.
- **Пресет боковой панели**: именованный вариант фона/градиента навигации, задаваемый пользователем.

### Constitution Alignment *(mandatory)*

- **ERP/domain fit**: Улучшает единообразие операционного интерфейса Phase 01 для снижения ошибок при переключении между корпоративными консолями; на бизнес-данные заказов/производства не влияет.
- **TOC readiness**: Не изменяет расчёт потоков, буферов и очередей; косвенно поддерживает скорость работы за счёт предсказуемого UI.
- **Traceability/audit**: Не требует новых бизнес-событий аудита; изменения носят презентационный характер.
- **Security/API boundary**: Не меняет границы API и модель авторизации; только клиентское представление.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Не менее 100% позиций утверждённой матрицы соответствия для тёмной темы имеют статус «соответствует» или «отступление согласовано» до выпуска изменений.
- **SC-002**: Не менее 100% проверок контраста для зафиксированных в матрице типовых пар «текст/фон» и «контрол/фон» в тёмной теме удовлетворяют WCAG 2.1 AA.
- **SC-003**: Регрессионная проверка по всем поддерживаемым пресетам боковой панели в тёмной теме завершается без блокирующих дефектов (0 блокирующих) по заранее определённому чеклисту.
- **SC-004**: Не менее 90% участников внутреннего UX/дизайн ревью подтверждают, что тёмная тема «достаточно согласована с эталоном» при сохранении узнаваемости CTfind (шкала или бинарный опрос — фиксируется в плане проверки).
- **SC-005**: Переключение темы в тестовом сценарии из пяти последовательных циклов не выявляет устойчивых визуальных артефактов по критериям QA (зафиксированным заранее).

## Assumptions

- Эталонная реализация доступна для аудита по согласованным путям в смежном репозитории (см. ниже); изменения в эталоне вне контроля данной фичи фиксируются версией снимка матрицы.
- Компонентная библиотека кабинета остаётся текущей (Tailwind/shadcn-подобные паттерны); семантика PrimeVue из эталона переносится по смыслу, а не копированием классов.
- Список экранов для обязательного QA будет сокращён до репрезентативного набора (shell, навигация, топбар, типовая таблица, типовая форма, модальное окно, один-два доменных экрана), если полный охват всех маршрутов не оговорён отдельно.
- Светлая тема не выравнивается с PAM в этом инкременте.
