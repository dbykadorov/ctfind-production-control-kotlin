# Research: PAM UC dark theme alignment (015)

## R-001 — Стратегия маппинга палитры

**Decision**: Семантическое выравнивание с эталоном PAM (`_vars.scss` для `:root[data-theme="dark"]`), а не пиксельное копирование каждого экрана. Значения переносятся в токены кабинета (`--bg-app`, `--bg-app-grad-*`, `--c-fg*`, `--c-border*`, `--c-overlay`, статусные `--c-status-*`, поверхности внутри/вне `.cabinet-card`) по таблице в `contracts/theme-pam-alignment.contract.md`.

**Rationale**: Спецификация (FR-002, FR-003) требует сохранить amber-бренд и elevated-card модель; эталон PAM использует cyan primary и иную композицию экранов — полное визуальное совпадение исключено.

**Alternatives considered**:

- Ввести полную дублирующую шкалу `--theme100`…`--theme800` в кабинете — отклонено как MVP: усложняет Tailwind-мост без явного запроса в спеке; можно добавить позже как алиасы при согласовании с дизайном.
- Подключать SCSS из репозитория PAM — отклонено: другой стек и PrimeVue, не поддерживается конституцией проекта.

## R-002 — Акцент и карточка контента

**Decision**: Primary interactive accent остаётся amber (`--c-brand-*`). Эталонный `--primary-color` (cyan PAM) не заменяет бренд CTfind; допускается только для вторичных декоративных элементов, если матрица это фиксирует как отступление. `.cabinet-card` сохраняется как elevated-island по форме/тени, но её surface в dark синхронизируется с тёмной PAM-ступенью.

**Rationale**: FR-002, FR-003 и User Story 2.

## R-003 — Оверлей модалок

**Decision**: Привести `--c-overlay` / использование затемнения в модалках к политике эталона по смыслу (`--overlay` в PAM dark: `rgb(0 0 0 / 60%)`), с допуском ±непрозрачности, зафиксированным в контракте.

**Rationale**: FR-007; текущий кабинет уже использует rgba через токены и Teleport для части модалок — проверить единообразие.

## R-004 — Подавление transition при смене темы

**Decision**: Реализовать аналог PAM `.theme-switching`: на время применения `data-theme` на `<html>` (или корневом элементе, где задаётся тема) выставлять класс, который через CSS отключает transitions для `background-color`, `color`, `border-color`, `fill`, `stroke`, `box-shadow` на 0s, затем снимать класс после `requestAnimationFrame` (двойной кадр) или короткого `setTimeout(0)`.

**Rationale**: FR-008; глобальные transitions в `globals.css` для html/body могут конфликтовать с мгновенной сменой темы.

**Alternatives**: Полностью убрать transition цветов с html/body — отклонено из‑за желания сохранить плавную смену темы при других сценариях.

## R-005 — Пресеты sidebar и login

**Decision**: Все пресеты из `CabinetSidebarPreset` проверяются вручную в dark после изменения токенов (SC-003). Страница логина включается в QA-маршруты или явно помечается исключением в матрице — см. `contracts/qa-signoff-routes.contract.md`.

**Rationale**: Edge cases в spec (градиентные пресеты, login).

## R-006 — Верификация контраста

**Decision**: Для SC-002 использовать зафиксированный список пар «элемент / фон» в матрице и проверку через DevTools / axe / WebAIM по каждому релизу фичи; автоматизацию (Playwright + контраст) оставить опциональным follow-up.

**Rationale**: Фича преимущественно токенная; автотесты контраста на всём DOM дороги в первом инкременте.
