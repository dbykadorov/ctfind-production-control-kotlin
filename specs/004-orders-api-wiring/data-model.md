# Data Model: Orders API + Frontend Wiring

## Customer

Represents an existing company or person placing orders.

**Fields**:

- `id: UUID` - stable internal identifier
- `displayName: String` - required business display name
- `status: CustomerStatus` - `ACTIVE` or `INACTIVE`
- `contactPerson: String?`
- `phone: String?`
- `email: String?`
- `createdAt: Instant`
- `updatedAt: Instant`

**Validation**:

- `displayName` must be non-blank and unique enough for search display.
- Only `ACTIVE` customers can be selected for new orders.
- Existing orders may continue referencing inactive customers.

**Relationships**:

- One customer can have many customer orders.

## CustomerOrder

Represents a sales/production order tied to a customer.

**Fields**:

- `id: UUID`
- `orderNumber: String` - unique human-readable number generated by the backend
- `customerId: UUID`
- `deliveryDate: LocalDate`
- `status: OrderStatus`
- `notes: String?`
- `createdByUserId: UUID`
- `createdAt: Instant`
- `updatedAt: Instant`
- `version: Long` - optimistic concurrency token

**Validation**:

- `orderNumber` is required, unique, and generated server-side.
- `customerId` must reference an existing customer.
- Creation requires an active customer.
- `deliveryDate` is required.
- New orders always start as `NEW`.
- Non-admin regular field edits are forbidden after `SHIPPED`.
- Update and status requests must include the expected `version`.

**Relationships**:

- Many orders belong to one customer.
- One order has one or more order items.
- One order has zero or more status changes.
- One order has zero or more order change diffs.

## CustomerOrderItem

Represents one line in an order.

**Fields**:

- `id: UUID`
- `orderId: UUID`
- `lineNo: Int`
- `itemName: String`
- `quantity: BigDecimal`
- `uom: String`
- `createdAt: Instant`
- `updatedAt: Instant`

**Validation**:

- `lineNo` must be positive and unique inside the order.
- `itemName` must be non-blank.
- `quantity` must be greater than zero.
- `uom` must be non-blank.
- An order must have at least one item.

**Relationships**:

- Many order items belong to one order.

## OrderStatus

Lifecycle state for an order.

**Values**:

- `NEW` - user-facing label: `новый`
- `IN_WORK` - user-facing label: `в работе`
- `READY` - user-facing label: `готов`
- `SHIPPED` - user-facing label: `отгружен`

**Allowed transitions**:

```text
NEW -> IN_WORK -> READY -> SHIPPED
```

**Forbidden transitions**:

- Reverse transitions.
- Skipping stages.
- Reapplying the current status as a business transition.

## OrderStatusChange

Represents one lifecycle transition.

**Fields**:

- `id: UUID`
- `orderId: UUID`
- `fromStatus: OrderStatus?`
- `toStatus: OrderStatus`
- `actorUserId: UUID`
- `changedAt: Instant`
- `note: String?`

**Validation**:

- `toStatus` is required.
- For status transition records after creation, `fromStatus` is required.
- Transition must satisfy the order status policy.
- `note` is optional and bounded to a reasonable UI length.

**Relationships**:

- Many status changes belong to one order.
- Actor references an authenticated user.

## OrderChangeDiff

Represents key business-field changes for order updates.

**Fields**:

- `id: UUID`
- `orderId: UUID`
- `actorUserId: UUID`
- `changedAt: Instant`
- `changeType: OrderChangeType` - `CREATED`, `UPDATED`, or `STATUS_CHANGED`
- `fieldDiffs: Json` - array/object of field names and before/after values
- `beforeSnapshot: Json?` - reserved for future full before snapshot
- `afterSnapshot: Json?` - reserved for future full after snapshot

**Validation**:

- Successful order edits must create at least one diff entry for changed key business fields.
- `fieldDiffs` must include only user-reviewable fields in Phase 1: customer, delivery date, notes, item summary, and status when applicable.
- Snapshot columns may remain null in Phase 1 but must not require schema replacement later.

**Relationships**:

- Many change diffs belong to one order.
- Actor references an authenticated user.

## AuditEvent

Business audit record for significant actions.

**Fields**:

- `id: UUID`
- `eventType: String` - e.g. `ORDER_CREATED`, `ORDER_UPDATED`, `ORDER_STATUS_CHANGED`
- `actorUserId: UUID`
- `targetType: String` - `CUSTOMER_ORDER`
- `targetId: UUID`
- `eventAt: Instant`
- `summary: String`
- `metadata: Json?`

**Validation**:

- Every successful create, update, and status change creates an audit event.
- Forbidden and invalid writes must not mutate order data.

## Query Models

### OrderListFilter

- `search: String?` - matches order number or customer display name
- `status: OrderStatus?`
- `customerId: UUID?`
- `activeOnly: Boolean?` - non-shipped orders
- `overdueOnly: Boolean?` - delivery date before today and not shipped
- `deliveryDateFrom: LocalDate?`
- `deliveryDateTo: LocalDate?`
- `page: Int`
- `size: Int`
- `sort: String?`

### DashboardSummary

- `totalOrders: Long`
- `activeOrders: Long`
- `overdueOrders: Long`
- `statusCounts: Map<OrderStatus, Long>`
- `recentChanges: List<OrderStatusChangeSummary>`
- `trend: List<OrderTrendPoint>`

## State And Permission Rules

- Any authenticated Phase 1 role can list and view orders read-only.
- Only `ADMIN` and `ORDER_MANAGER` can create orders, edit non-shipped orders, and change status.
- Unauthenticated requests return `401`.
- Authenticated users without write permission receive `403` for write/status endpoints.
- Stale updates return `409` and leave data unchanged.
- Invalid validation or lifecycle transitions return `400` or `422` and leave data unchanged.
