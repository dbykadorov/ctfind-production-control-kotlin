# Feature Specification: Инфраструктура и модель внутренних уведомлений

**Feature Branch**: `008-notifications-infrastructure`
**Created**: 2026-04-28
**Status**: Draft
**Input**: User description: "Инфраструктура и модель внутренних уведомлений (M7, Phase 1). Спека 1 из 3: домен Notification, Flyway миграция, порт NotificationCreatePort для записи из других модулей, REST API для чтения/управления уведомлениями. Все аутентифицированные роли видят только свои уведомления. Только in-app, без Telegram/email."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Пользователь читает свои уведомления (Priority: P1)

Аутентифицированный пользователь любой роли (администратор, менеджер заказов, мастер цеха, исполнитель) обращается к системе за списком своих уведомлений. Система возвращает уведомления, адресованные именно ему, отсортированные от новых к старым, с постраничной навигацией. Каждое уведомление содержит тип события, заголовок, текст, привязку к объекту (заказ или производственная задача, если применимо) и признак «прочитано / не прочитано». Пользователь также может запросить только непрочитанные уведомления. Отдельная лёгкая операция возвращает количество непрочитанных уведомлений — она предназначена для периодического опроса из интерфейса.

**Why this priority**: Это минимально жизнеспособный функционал уведомлений: без возможности прочитать уведомления все остальные части (триггеры, UI) бессмысленны. Этот сценарий формирует контракт хранения и чтения.

**Independent Test**: Под любым аутентифицированным пользователем (например, `admin / admin`) выполнить запрос списка уведомлений. При наличии уведомлений в базе — получить список с полями type, title, body, targetType, targetId, read, createdAt, разбитый на страницы. При пустой базе — получить пустую страницу с totalItems=0. Запрос количества непрочитанных возвращает число >= 0.

**Acceptance Scenarios**:

1. **Given** у пользователя есть 5 уведомлений (3 непрочитанных, 2 прочитанных), **When** пользователь запрашивает список уведомлений без фильтров, **Then** получает страницу с 5 записями, отсортированными по времени создания по убыванию, каждая содержит id, type, title, body, targetType, targetId, read, createdAt.
2. **Given** у пользователя есть 3 непрочитанных и 2 прочитанных уведомления, **When** пользователь запрашивает только непрочитанные, **Then** получает страницу с 3 записями, все с признаком read=false.
3. **Given** у пользователя есть 60 уведомлений, размер страницы 20, **When** пользователь запрашивает 2-ю страницу, **Then** получает записи 21–40, totalItems=60, totalPages=3.
4. **Given** у пользователя есть 3 непрочитанных уведомления, **When** пользователь запрашивает количество непрочитанных, **Then** получает { count: 3 }.
5. **Given** у пользователя A есть уведомления, а у пользователя B — нет, **When** пользователь B запрашивает список, **Then** получает пустую страницу, а уведомления A ему не видны.
6. **Given** неаутентифицированный запрос, **When** выполняется обращение к любому эндпоинту уведомлений, **Then** система возвращает 401.

### User Story 2 — Пользователь управляет прочитанностью уведомлений (Priority: P2)

Пользователь хочет пометить конкретное уведомление как прочитанное (например, после клика по нему в будущем UI) или пометить все непрочитанные разом (кнопка «Прочитать все»). Операция пометки одного уведомления идемпотентна — повторный вызов не вызывает ошибки. Массовая операция возвращает количество фактически обновлённых записей.

**Why this priority**: Управление прочитанностью — второй по важности сценарий после чтения. Без него badge с непрочитанными будет вечно расти, а уведомления останутся бесполезным шумом. Зависит от P1 (модель и хранение).

**Independent Test**: Создать уведомление для пользователя, затем пометить его прочитанным — убедиться, что read=true. Повторить — убедиться, что ошибки нет. Вызвать «прочитать все» — убедиться, что все непрочитанные стали read=true и ответ содержит корректный счётчик.

**Acceptance Scenarios**:

1. **Given** у пользователя есть непрочитанное уведомление с id X, **When** пользователь помечает его прочитанным, **Then** уведомление X имеет read=true.
2. **Given** уведомление X уже прочитано, **When** пользователь повторно помечает его прочитанным, **Then** ответ успешен (идемпотентно), состояние не меняется.
3. **Given** пользователь A пытается пометить прочитанным уведомление пользователя B, **When** выполняется запрос, **Then** система возвращает 404 (не раскрывая существование чужого уведомления).
4. **Given** пользователь пытается пометить несуществующее уведомление, **When** выполняется запрос, **Then** система возвращает 404.
5. **Given** у пользователя есть 5 непрочитанных и 3 прочитанных уведомления, **When** пользователь вызывает «прочитать все», **Then** все 5 непрочитанных становятся read=true, ответ содержит { updated: 5 }.
6. **Given** у пользователя нет непрочитанных уведомлений, **When** пользователь вызывает «прочитать все», **Then** ответ содержит { updated: 0 }.

### User Story 3 — Другой модуль создаёт уведомление через порт (Priority: P1)

Модуль производственных задач (или заказов) должен иметь возможность создать уведомление для конкретного пользователя, указав тип события, заголовок, текст, привязку к объекту. Порт создания принимает данные и сохраняет уведомление в хранилище. Это внутренний контракт модуля — он не экспонируется наружу через REST API.

**Why this priority**: P1 наравне со чтением, потому что без возможности записи уведомлений модуль бесполезен. Спека 2 (триггеры) будет вызывать этот порт.

**Independent Test**: Вызвать порт создания программно (из теста), передав корректные данные — убедиться, что уведомление сохранено и доступно через API чтения. Вызвать с невалидными данными — убедиться, что срабатывает валидация домена.

**Acceptance Scenarios**:

1. **Given** корректные данные уведомления (recipientUserId, type, title, targetType, targetId), **When** вызывается порт создания, **Then** уведомление сохраняется с read=false, createdAt установлен системой, и доступно через GET /api/notifications для получателя.
2. **Given** данные без обязательного поля title (пустая строка), **When** вызывается порт создания, **Then** операция отклоняется с ошибкой валидации.
3. **Given** title длиной 201 символ, **When** вызывается порт создания, **Then** операция отклоняется с ошибкой валидации.
4. **Given** body длиной 1001 символ, **When** вызывается порт создания, **Then** операция отклоняется с ошибкой валидации.
5. **Given** уведомление без привязки к объекту (targetType=null, targetId=null), **When** вызывается порт создания, **Then** уведомление успешно сохраняется — привязка к объекту необязательна.

### Edge Cases

- Пользователь запрашивает страницу за пределами диапазона (page=100, а всего 2 страницы): система возвращает пустую страницу с корректным totalItems/totalPages, а не ошибку.
- Параметр size > 100: система ограничивает его значением 100 без ошибки (silent clamp).
- Параметр size <= 0 или page < 0: система использует значения по умолчанию (page=0, size=20).
- Одновременные запросы mark-all-read от одного пользователя: операция идемпотентна, не приводит к ошибке или двойному обновлению.
- Уведомление с targetId, указывающим на удалённый или недоступный объект: уведомление хранится и отдаётся as-is — валидация существования объекта не входит в ответственность модуля уведомлений.
- Очень длинный body (ровно 1000 символов): допускается, 1001 — отклоняется.
- Тип уведомления, не предусмотренный текущим enum: enum хранится в БД как строка (VARCHAR), поэтому новые типы из спеки 2 добавляются без миграции схемы.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять хранилище уведомлений с сущностью Notification, содержащей обязательные поля: идентификатор, получатель, тип события, заголовок, признак прочитанности, время создания; и опциональные: текст, тип привязанного объекта, идентификатор привязанного объекта.
- **FR-002**: Система MUST обеспечивать создание уведомлений через внутренний программный интерфейс (порт), доступный другим модулям приложения без прямого доступа к хранилищу уведомлений.
- **FR-003**: Система MUST предоставлять операцию получения списка уведомлений текущего пользователя с постраничной навигацией (размер страницы по умолчанию 20, максимум 100) и сортировкой от новых к старым.
- **FR-004**: Система MUST поддерживать фильтрацию списка уведомлений по признаку «только непрочитанные».
- **FR-005**: Система MUST предоставлять операцию получения количества непрочитанных уведомлений текущего пользователя — лёгкую и оптимизированную для частого вызова.
- **FR-006**: Система MUST предоставлять операцию пометки одного уведомления текущего пользователя как прочитанного. Операция MUST быть идемпотентной.
- **FR-007**: Система MUST предоставлять операцию массовой пометки всех непрочитанных уведомлений текущего пользователя как прочитанных, возвращая количество фактически обновлённых записей.
- **FR-008**: Система MUST гарантировать изоляцию данных: пользователь видит и управляет только уведомлениями, адресованными ему. Попытка обратиться к чужому уведомлению MUST возвращать состояние «не найдено», не раскрывая факт существования.
- **FR-009**: Все операции с уведомлениями MUST требовать аутентификацию. Неаутентифицированные запросы MUST получать ответ 401.
- **FR-010**: Тип уведомления MUST храниться способом, позволяющим добавлять новые типы без изменения схемы хранения.
- **FR-011**: Заголовок уведомления MUST быть непустым и не превышать 200 символов. Текст уведомления, если указан, MUST не превышать 1000 символов.
- **FR-012**: Привязка к объекту (тип объекта + идентификатор) MUST быть опциональной. Уведомления без привязки к объекту допускаются.

### Key Entities

- **Notification**: Единица уведомления, адресованная конкретному пользователю. Ключевые атрибуты: уникальный идентификатор, получатель (ссылка на пользователя системы), тип события (расширяемое перечисление), заголовок (обязательный, до 200 символов), текст (опциональный, до 1000 символов), тип привязанного объекта (ORDER или PRODUCTION_TASK, опционально), идентификатор привязанного объекта (человекочитаемый — имя заказа или номер задачи, опционально), признак прочитанности (по умолчанию «не прочитано»), время создания.

### Constitution Alignment *(mandatory)*

- **ERP/domain fit**: Уведомления — новый операционный модуль, укрепляющий связь между производственными задачами, заказами и пользователями. Это часть M7 «Внутренние уведомления» из Phase 1.
- **TOC readiness**: Уведомления сохраняют факт «когда пользователь узнал о событии» (createdAt) и «когда отреагировал» (момент прочтения) — это данные о времени реакции, полезные для будущего анализа задержек и узких мест. targetType/targetId связывают уведомление с потоком задач.
- **Traceability/audit**: Создание уведомления фиксирует факт информирования пользователя о событии. Сами уведомления не подлежат удалению в Phase 1, что обеспечивает сохранность истории информирования.
- **Security/API boundary**: Все эндпоинты защищены JWT-аутентификацией. Изоляция данных — строгая: пользователь оперирует только своими уведомлениями. Чужие уведомления возвращаются как «не найдено» (404), а не «запрещено» (403), чтобы не раскрывать существование. Уведомления доступны всем ролям (ADMIN, ORDER_MANAGER, PRODUCTION_SUPERVISOR, PRODUCTION_EXECUTOR) — это не привилегированная функция.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Любой аутентифицированный пользователь может получить список своих уведомлений за время, не заметное при ручном взаимодействии (ожидание — менее 1 секунды для страницы из 20 записей при объёме до 10 000 уведомлений на пользователя).
- **SC-002**: Запрос количества непрочитанных уведомлений выполняется быстро и стабильно при частом вызове (ожидание — пригоден для опроса каждые 30 секунд без заметного влияния на систему).
- **SC-003**: Пользователь никогда не видит уведомления, адресованные другому пользователю — 100% изоляция данных.
- **SC-004**: Новые типы уведомлений (из спеки 2) добавляются без изменений схемы хранения — только код.
- **SC-005**: Модуль уведомлений работает автономно: его можно протестировать и запустить независимо от модулей заказов и производственных задач.

## Assumptions

- Канал доставки — только внутри системы (in-app). Telegram, email и push-уведомления — вне scope Phase 1 (§5 PHASE_01.md).
- Удаление уведомлений — вне scope. Пользователь может только пометить уведомление прочитанным.
- Настройки предпочтений уведомлений (отключение типов, частота) — вне scope Phase 1. Все типы уведомлений включены для всех пользователей.
- Retention/cleanup старых уведомлений — вне scope. Может быть добавлен позже как scheduled job.
- WebSocket/SSE для push-доставки — вне scope. Фронтенд (спека 3) будет использовать polling.
- Batch-операции, кроме «прочитать все», — вне scope.
- Порт создания уведомлений — синхронный. Асинхронная очередь (event bus) — вне scope Phase 1.
- Существующая таблица `app_user` используется как источник получателей; проверка существования пользователя при создании уведомления — ответственность вызывающего модуля.
