# Implementation Plan: Склад — материалы и остатки

**Branch**: `011-warehouse-materials` | **Date**: 2026-04-29 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/011-warehouse-materials/spec.md`

## Summary

Новый модуль `inventory` в гексагональной архитектуре: CRUD материалов с фиксированными единицами измерения, ручной приход с журналом движений, вычисляемые остатки (SUM), аудит. Роль WAREHOUSE. Frontend: страница списка, карточка материала с движениями, модальные окна создания/редактирования/прихода, Sidebar-пункт.

## Technical Context

**Language/Version**: Kotlin / Java 21 (backend), TypeScript strict (frontend)
**Primary Dependencies**: Spring Boot, JPA/Hibernate, Flyway (backend); Vue 3, Pinia, vue-router, Tailwind, vue-i18n, lucide-vue-next (frontend)
**Storage**: PostgreSQL (Flyway migration V8)
**Testing**: JUnit 5 + MockK (backend); Vitest source-text TDD (frontend)
**Target Platform**: Docker Compose (Linux server), SPA in browser
**Project Type**: Web application (modular monolith backend + Vue 3 SPA)
**Performance Goals**: Standard web app — list of materials loads < 1s for up to 1000 items
**Constraints**: Up to 50 concurrent users (Phase 1 scope)
**Scale/Scope**: ~15 backend files, ~15 frontend files, 1 migration

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- **ERP domain fit**: Inventory/warehouse is a core ERP entity listed in Phase 1 scope (PHASE_01.md §4.4). Closes the last unimplemented role ("Склад") and screen ("Склад") from the product spec.
- **Constraint-aware operations**: Stock movements preserve full history (who, when, how much, why). Each movement is immutable with timestamps. No FIFO/priority hardcoded — future TOC analysis can query movement history for lead times and buffer calculations.
- **Architecture boundaries**: Business rules (name uniqueness, deletion guard, quantity validation) live in domain/application. Controllers only adapt HTTP ↔ use case. JPA entities isolated in persistence adapter. Frontend handles display only.
- **Traceability/audit**: All CRUD operations + receipts write to `inventory_audit_event` table, integrated into the unified audit log via new `AuditCategory.INVENTORY`. Actor identity preserved.
- **API-only/security**: Backend is REST API with JWT auth. New role `WAREHOUSE` added via migration. Permission checks in use cases (not controllers). Frontend uses existing `isWarehouse` flag from `use-permissions.ts`.
- **Docker/verifiability**: Flyway migration V8 auto-runs on docker-compose up. Verification: `make backend-test-docker`, `pnpm typecheck && pnpm test && pnpm build`, `make docker-up-detached && make health`, API smoke test from quickstart.md.
- **Exception handling**: No violations — standard hexagonal module following orders/production patterns.
