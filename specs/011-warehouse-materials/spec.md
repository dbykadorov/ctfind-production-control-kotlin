# Feature Specification: Склад — материалы и остатки

**Feature Branch**: `011-warehouse-materials`  
**Created**: 2026-04-29  
**Status**: Draft  
**Input**: Завершающий модуль Phase 1. Упрощённый складской учёт: CRUD материалов, ручной приход, текущие остатки. Роль WAREHOUSE_MANAGER. Без списания (спека 2), без артикулов, без партий.

## User Scenarios & Testing

### User Story 1 — Справочник материалов (Priority: P1)

Кладовщик открывает раздел «Склад» и видит таблицу всех материалов с текущими остатками. Он может добавить новый материал (указав название и единицу измерения), отредактировать существующий или удалить материал, у которого нет движений.

**Why this priority**: Без справочника материалов невозможны никакие складские операции. Это фундамент модуля.

**Independent Test**: Войти как пользователь с ролью «Склад» → увидеть пункт «Склад» в боковом меню → открыть → увидеть пустой список → создать материал «Фанера» (м²) → увидеть его в списке с остатком 0 → отредактировать название → удалить материал.

**Acceptance Scenarios**:

1. **Given** пользователь с ролью WAREHOUSE_MANAGER или Administrator авторизован, **When** он открывает /cabinet/warehouse, **Then** отображается таблица материалов с колонками: название, единица измерения, остаток
2. **Given** список материалов открыт, **When** пользователь нажимает «Добавить материал» и заполняет форму (название, ед.изм.), **Then** материал появляется в таблице с остатком 0
3. **Given** материал существует и не имеет движений, **When** пользователь удаляет его, **Then** материал исчезает из списка
4. **Given** материал имеет хотя бы одно движение (приход), **When** пользователь пытается удалить его, **Then** система показывает ошибку «Невозможно удалить материал с историей движений»
5. **Given** существует 30+ материалов, **When** пользователь вводит текст в поле поиска, **Then** таблица фильтруется по названию материала
6. **Given** пользователь без роли WAREHOUSE_MANAGER или Administrator, **When** он пытается открыть /cabinet/warehouse, **Then** система показывает страницу «Доступ запрещён»

---

### User Story 2 — Приход материала (Priority: P1)

Кладовщик регистрирует поступление материала на склад: выбирает материал из списка, вводит количество и опциональный комментарий. Каждый приход сохраняется как отдельная запись в журнале движений. Текущий остаток материала обновляется автоматически.

**Why this priority**: Приход — единственная операция изменения остатков в этой спеке. Без неё остатки всегда 0, модуль бесполезен.

**Independent Test**: Создать материал → оформить приход 100 шт → увидеть остаток 100 → оформить ещё приход 50 шт → увидеть остаток 150 → открыть журнал движений → увидеть 2 записи.

**Acceptance Scenarios**:

1. **Given** материал «Фанера» с остатком 0 существует, **When** кладовщик оформляет приход 100 м², **Then** остаток материала становится 100, в журнале движений появляется запись с типом «Приход», количеством 100 и датой
2. **Given** материал «Фанера» с остатком 100, **When** кладовщик оформляет приход 50 с комментарием «Поставка от ООО Лес», **Then** остаток становится 150, запись движения содержит комментарий
3. **Given** форма прихода открыта, **When** кладовщик вводит количество 0 или отрицательное число, **Then** кнопка подтверждения неактивна, показана ошибка валидации
4. **Given** материал «Фанера» существует, **When** кладовщик открывает журнал движений по этому материалу, **Then** отображается список всех приходов в обратном хронологическом порядке с полями: дата, тип, количество, комментарий, кто оформил

---

### User Story 3 — Навигация и боковое меню (Priority: P2)

Пункт «Склад» появляется в боковом меню для пользователей с ролью WAREHOUSE_MANAGER и Administrator. Пункт ведёт на /cabinet/warehouse.

**Why this priority**: Навигация — вспомогательная функция, маршрут можно открыть напрямую по URL.

**Independent Test**: Войти как «Склад» → увидеть пункт «Склад» в меню → кликнуть → попасть на /cabinet/warehouse.

**Acceptance Scenarios**:

1. **Given** пользователь с ролью WAREHOUSE_MANAGER авторизован, **When** он смотрит боковое меню, **Then** пункт «Склад» отображается с иконкой
2. **Given** пользователь с ролью Administrator авторизован, **When** он смотрит боковое меню, **Then** пункт «Склад» отображается
3. **Given** пользователь с ролью ORDER_MANAGER (без WAREHOUSE_MANAGER), **When** он смотрит боковое меню, **Then** пункт «Склад» не отображается

---

### User Story 4 — Аудит складских операций (Priority: P2)

Все складские операции (создание материала, редактирование, удаление, приход) фиксируются в общем журнале аудита системы. Администратор может увидеть складские события в разделе «Журнал действий».

**Why this priority**: Аудит важен для прозрачности, но модуль функционален и без него.

**Independent Test**: Создать материал → оформить приход → открыть журнал аудита как администратор → увидеть записи о создании материала и приходе.

**Acceptance Scenarios**:

1. **Given** кладовщик создал материал «Фанера», **When** администратор открывает журнал аудита, **Then** видна запись с категорией «Склад», типом «MATERIAL_CREATED», именем актора и описанием
2. **Given** кладовщик оформил приход 100 м² фанеры, **When** администратор открывает журнал аудита, **Then** видна запись с типом «STOCK_RECEIPT», количеством и комментарием в описании

---

### Edge Cases

- Что происходит при попытке создать материал с дублирующимся названием? — Система отклоняет с ошибкой «Материал с таким названием уже существует»
- Что происходит при вводе очень большого числа в приходе (например, 999999999)? — Система принимает, ограничение на уровне числового типа (BigDecimal)
- Что если два кладовщика одновременно оформляют приход одного материала? — Оба прихода сохраняются, остаток корректен (каждый приход — отдельная запись, остаток вычисляется как сумма)
- Что если название материала содержит спецсимволы? — Допускается, но лидирующие/трейлинг пробелы обрезаются
- Может ли остаток стать отрицательным? — Нет, в этой спеке нет операций уменьшения остатка (списание — спека 2)

## Requirements

### Functional Requirements

- **FR-001**: Система MUST позволять создавать материал с обязательными полями: название (уникальное) и единица измерения (выбор из фиксированного списка)
- **FR-002**: Система MUST отображать список материалов в виде таблицы с колонками: название, единица измерения, текущий остаток
- **FR-003**: Система MUST поддерживать поиск материалов по названию (фильтрация по подстроке, без учёта регистра)
- **FR-004**: Система MUST позволять редактировать название и единицу измерения существующего материала
- **FR-005**: Система MUST позволять удалять материал только если у него нет записей движений
- **FR-006**: Система MUST позволять оформлять приход материала: выбор материала, ввод количества (строго > 0), опциональный комментарий
- **FR-007**: Система MUST сохранять каждый приход как отдельную запись движения с полями: материал, количество, комментарий, кто оформил, дата/время
- **FR-008**: Система MUST вычислять текущий остаток материала как сумму всех приходов по этому материалу
- **FR-009**: Система MUST отображать журнал движений по конкретному материалу в обратном хронологическом порядке
- **FR-010**: Система MUST ограничивать доступ к модулю складу ролями WAREHOUSE_MANAGER и Administrator
- **FR-011**: Система MUST фиксировать в журнале аудита: создание, редактирование, удаление материала и каждый приход
- **FR-012**: Система MUST поддерживать пагинацию в списке материалов и журнале движений
- **FR-013**: Единицы измерения — фиксированный набор: шт, кг, м, л, м², м³
- **FR-014**: Система MUST отображать пункт «Склад» в боковом меню только для пользователей с ролью WAREHOUSE_MANAGER или Administrator

### Key Entities

- **Материал (Material)**: основная сущность — название (уникальное, строка), единица измерения (enum из фиксированного набора). Текущий остаток — вычисляемое значение, не хранится отдельно.
- **Движение материала (StockMovement)**: запись о приходе — связь с материалом, количество (положительное число), тип операции (в этой спеке только RECEIPT), комментарий, кто оформил (связь с пользователем), дата/время.

### Constitution Alignment

- **ERP/domain fit**: Модуль «Склад» — последний компонент Phase 1, закрывает роль «Склад» из ролевой модели. Учёт материалов и остатков — базовая ERP-функция.
- **TOC readiness**: Остатки материалов — информация о доступности ресурсов для производства. В спеке 2 (списание под заказ) это станет частью потока заказ→производство→материалы.
- **Traceability/audit**: Все операции (CRUD материалов + приход) пишутся в общий аудит-лог. Каждое движение хранит актора и timestamp.
- **Security/API boundary**: Новая роль WAREHOUSE_MANAGER. Доступ к API /api/materials ограничен этой ролью + Administrator. JWT-аутентификация как у всех эндпоинтов.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Кладовщик может создать новый материал и оформить первый приход менее чем за 1 минуту
- **SC-002**: Текущий остаток материала корректно отражает сумму всех приходов в любой момент времени
- **SC-003**: Поиск по названию возвращает результаты при наличии 100+ материалов без заметной задержки
- **SC-004**: Администратор видит все складские события в журнале аудита с корректной атрибуцией (кто, когда, что)
- **SC-005**: Пользователь без складской роли не может получить доступ к складским данным ни через UI, ни через API

## Assumptions

- Роль «Warehouse» (WAREHOUSE_MANAGER) уже определена в системе ролей и доступна в use-permissions composable (isWarehouse)
- Единицы измерения — фиксированный enum, расширение набора — ручное изменение кода
- В этой спеке нет операций уменьшения остатка — остаток может только расти или оставаться неизменным
- Списание материалов под заказ — scope спеки 012
- Количество в приходе — дробное число (BigDecimal), для поддержки кг, м и т.д.
- Справочник материалов общий для всей системы, не привязан к конкретному складу или локации
