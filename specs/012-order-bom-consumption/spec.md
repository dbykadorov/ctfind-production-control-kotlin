# Feature Specification: Списание материалов под заказ + BOM

**Feature Branch**: `012-order-bom-consumption`
**Created**: 2026-04-29
**Status**: Accepted
**Input**: Продолжение Phase 1 / inventory-модуля. Спека 011 (`011-warehouse-materials`) добавила материалы и приходы. Теперь нужно подключить материалы к заказам: указывать в заказе спецификацию материалов (BOM), списывать материал со склада с привязкой к заказу, видеть фактический расход и остаток к списанию.

## Clarifications

### Session 2026-04-29

- Q: Должны ли Production Supervisor и Executor видеть секции «Материалы (BOM)» и «Расход материалов» в карточке заказа? → A: Production Supervisor — да, Executor — нет.
- Q: Должна ли спека включать автоматическую отправку уведомления о перерасходе материала по заказу? → A: Нет. Перерасход виден только в UI карточки заказа и в журнале аудита; уведомления — out of scope.
- Q: Что считать «активным заказом» для блокировки удаления материала из справочника? → A: Любой статус, кроме «Завершён» и «Отменён» — единая политика с правилами BOM/списания (FR-005, FR-012).

## User Scenarios & Testing

### User Story 1 — Спецификация материалов в заказе (BOM) (Priority: P1)

Менеджер заказа открывает карточку заказа и в разделе «Материалы» добавляет строки спецификации: какой материал и сколько потребуется для выполнения этого заказа. Каждая строка — отдельный материал из справочника склада с указанным количеством и опциональным комментарием. Строку можно отредактировать (изменить количество, комментарий) или удалить.

**Why this priority**: Без BOM невозможно ни планирование расхода, ни последующее списание. Это база, от которой зависят остальные истории.

**Independent Test**: Открыть существующий заказ → добавить строку BOM «Фанера 10 м²» → увидеть строку в таблице → отредактировать количество на 12 → удалить строку → таблица пустая.

**Acceptance Scenarios**:

1. **Given** заказ существует и пользователь имеет права на редактирование BOM, **When** он открывает карточку заказа, **Then** отображается секция «Материалы (BOM)» с таблицей (материал, требуется, комментарий) и кнопкой «Добавить материал»
2. **Given** секция BOM открыта, **When** пользователь нажимает «Добавить материал», выбирает «Фанера», вводит 10, добавляет комментарий «На основу», **Then** в таблице появляется строка «Фанера / 10 м² / На основу»
3. **Given** в BOM уже есть «Фанера», **When** пользователь пытается добавить «Фанеру» ещё раз, **Then** система отклоняет с ошибкой «Этот материал уже есть в спецификации заказа»
4. **Given** строка BOM существует, **When** пользователь нажимает «Редактировать» и меняет количество с 10 на 12, **Then** строка в таблице обновляется, материал и комментарий не меняются
5. **Given** пользователь вводит количество ≤ 0, **When** он пытается сохранить, **Then** кнопка подтверждения неактивна, показана ошибка валидации
6. **Given** строка BOM существует и по этому материалу не было списаний по этому заказу, **When** пользователь удаляет строку, **Then** строка исчезает
7. **Given** строка BOM существует и по этому материалу уже было хотя бы одно списание, **When** пользователь пытается удалить строку, **Then** система отклоняет с ошибкой «Невозможно удалить материал, по которому уже были списания»
8. **Given** заказ имеет статус «Завершён» или «Отменён», **When** пользователь пытается изменить BOM, **Then** кнопки добавления/редактирования/удаления неактивны или скрыты

### User Story 2 — Списание материала под заказ (Priority: P1)

Кладовщик физически выдаёт материал в производство и регистрирует это в системе как списание: выбирает заказ, выбирает материал из BOM этого заказа, вводит фактически выданное количество и опциональный комментарий. Текущий остаток материала на складе уменьшается на эту величину. Если остатка недостаточно — операция блокируется. Каждое списание — отдельная запись в журнале движений с привязкой к заказу.

**Why this priority**: Списание — главная операция, ради которой существует модуль. Без неё нельзя контролировать расход.

**Independent Test**: Создать материал «Фанера», оформить приход 50 м² → создать заказ, добавить в его BOM «Фанера 10 м²» → списать 5 м² по этому заказу → остаток материала становится 45 → списать ещё 4 → остаток 41, фактический расход по заказу 9 → попробовать списать 100 → блокировка.

**Acceptance Scenarios**:

1. **Given** заказ имеет в BOM «Фанера 10 м²», текущий остаток фанеры 50, **When** кладовщик списывает 5 м² по этому заказу с комментарием «Выдано в цех», **Then** в журнале движений появляется запись с типом «Списание», количеством 5, ссылкой на заказ и комментарием; остаток фанеры становится 45
2. **Given** остаток фанеры 3 м², **When** кладовщик пытается списать 5 м², **Then** операция блокируется с ошибкой «Недостаточный остаток на складе. Доступно: 3 м²»
3. **Given** у материала «Фанера» нет строки BOM в выбранном заказе, **When** кладовщик пытается выбрать «Фанеру» в форме списания по этому заказу, **Then** материал не доступен в списке выбора (показываются только материалы из BOM заказа)
4. **Given** заказ в статусе «Отменён» или «Завершён», **When** кладовщик пытается оформить списание по этому заказу, **Then** заказ не доступен в списке выбора заказов или операция отклоняется
5. **Given** в BOM «Фанера 10 м²», по этому заказу уже списано 8 м², **When** кладовщик списывает ещё 5 м² (итого 13, перерасход 3), **Then** операция выполняется, но в форме отображается предупреждение «Перерасход относительно BOM: +3 м²» (не блокирующее)
6. **Given** кладовщик вводит количество ≤ 0, **When** он пытается сохранить, **Then** кнопка подтверждения неактивна, показана ошибка валидации

### User Story 3 — Расход материалов по заказу (Priority: P1)

В карточке заказа в секции «Расход материалов» отображается агрегированная таблица: для каждой строки BOM показывается требуемое количество, фактически списанное количество, остаток к списанию и индикатор перерасхода. Это позволяет менеджеру и кладовщику в любой момент видеть, сколько ещё нужно выдать и где есть превышение.

**Why this priority**: Без отображения расхода BOM и списания превращаются в «слепые» операции; пользователь не сможет планировать дальнейшие выдачи.

**Independent Test**: BOM «Фанера 10 м², Гвозди 100 шт» → списать 4 м² фанеры и 30 шт гвоздей → таблица показывает «Фанера 10/4/6», «Гвозди 100/30/70» → списать ещё 8 м² фанеры (перерасход 2) → строка фанеры показывает «10/12/0» с индикатором перерасхода.

**Acceptance Scenarios**:

1. **Given** в BOM заказа две строки и по обеим были списания, **When** пользователь открывает секцию «Расход материалов», **Then** отображается таблица с колонками: материал, ед.изм., требуется (BOM), списано, остаток к списанию
2. **Given** по строке BOM списано меньше, чем требуется, **When** отображается строка, **Then** «остаток к списанию» = требуется − списано (положительное число)
3. **Given** по строке BOM списано столько же, сколько требуется, **When** отображается строка, **Then** «остаток к списанию» = 0, строка визуально помечена как закрытая
4. **Given** по строке BOM списано больше, чем требуется, **When** отображается строка, **Then** «остаток к списанию» = 0 и отображается индикатор перерасхода с величиной превышения
5. **Given** в BOM есть материал, по которому ни одного списания не было, **When** отображается строка, **Then** «списано» = 0, «остаток к списанию» = требуется
6. **Given** заказ ещё не имеет BOM, **When** пользователь открывает секцию, **Then** отображается пустое состояние с подсказкой «Добавьте материалы в BOM, чтобы начать учёт расхода»

### User Story 4 — Аудит операций с материалами заказа (Priority: P2)

Все изменения BOM (добавление, редактирование, удаление строки) и все списания фиксируются в общем журнале аудита системы под категорией «Склад». Администратор может увидеть, кто и когда менял спецификацию материалов или выдавал материал по конкретному заказу.

**Why this priority**: Аудит важен для разбора расхождений (например, «почему столько списали»), но модуль работоспособен и без него.

**Independent Test**: Добавить строку BOM → отредактировать количество → списать материал → открыть журнал аудита → увидеть три записи с категорией «Склад», описаниями операций и ссылками на заказ и материал.

**Acceptance Scenarios**:

1. **Given** менеджер добавил в BOM заказа N строку «Фанера 10 м²», **When** администратор открывает журнал аудита, **Then** видна запись с категорией «Склад», типом «BOM_LINE_ADDED», именем актора, упоминанием заказа N и материала «Фанера»
2. **Given** менеджер изменил количество в строке BOM с 10 на 12, **When** администратор открывает журнал аудита, **Then** видна запись с типом «BOM_LINE_UPDATED» и описанием изменения
3. **Given** менеджер удалил строку BOM, **When** администратор открывает журнал аудита, **Then** видна запись с типом «BOM_LINE_REMOVED»
4. **Given** кладовщик списал 5 м² фанеры по заказу N, **When** администратор открывает журнал аудита, **Then** видна запись с типом «STOCK_CONSUMPTION», количеством, материалом и ссылкой на заказ

---

### Edge Cases

- Что происходит, если материал, на который ссылается строка BOM, удалили из справочника? — В справочнике 011 материал нельзя удалить, если по нему есть движения; добавляется правило: материал также нельзя удалить, если он есть хоть в одной строке BOM активного заказа. **Активный заказ** = любой заказ в статусе, отличном от «Завершён» и «Отменён» (та же политика, что в FR-005 и FR-012). Если все заказы, в BOM которых упоминается материал, находятся в статусах «Завершён» или «Отменён», и по нему не было движений — материал можно удалить.
- Что если два кладовщика одновременно списывают один материал и сумма их списаний превышает остаток? — Система применяет проверку остатка на момент сохранения; кто пришёл вторым — получает ошибку «Недостаточный остаток».
- Что если менеджер уменьшает требуемое количество BOM ниже уже списанного? — Операция допускается; в секции «Расход» строка отобразится как перерасход.
- Что если в заказе нет BOM, но кладовщик хочет списать материал? — Списать без строки BOM нельзя: список материалов в форме списания строится из BOM выбранного заказа. Сначала добавляется строка в BOM, затем списывается. Это сознательное ограничение, чтобы избегать «слепых» расходов вне плана.
- Можно ли списать на заказ, у которого ещё нет ни одной строки BOM? — Нет, см. предыдущий пункт.
- Что если требуемое количество в BOM очень дробное (например, 0,0001)? — Допустимо, BigDecimal с 4 знаками после запятой как у движений и приходов в 011.

## Requirements

### Functional Requirements

#### BOM (спецификация материалов заказа)

- **FR-001**: Система MUST позволять добавлять в заказ строку спецификации материалов (BOM) с полями: ссылка на материал из справочника, количество (строго > 0), опциональный комментарий
- **FR-002**: Система MUST обеспечивать уникальность пары (заказ, материал) в BOM — один материал не может присутствовать в BOM одного заказа более одного раза
- **FR-003**: Система MUST позволять редактировать количество и комментарий существующей строки BOM; материал в строке менять нельзя (нужно удалить и добавить заново)
- **FR-004**: Система MUST позволять удалять строку BOM только если по этой паре (заказ, материал) ещё не было ни одного списания
- **FR-005**: Система MUST блокировать любые изменения BOM (добавление, редактирование, удаление) для заказов в статусах «Завершён» и «Отменён»
- **FR-006**: Система MUST отображать BOM заказа в виде таблицы в карточке заказа, в обратном хронологическом порядке добавления строк (новые сверху)

#### Списание материала

- **FR-007**: Система MUST позволять оформлять операцию «Списание» с обязательным указанием: материал, заказ, количество (строго > 0); опциональный комментарий
- **FR-008**: Система MUST разрешать списание материала по заказу только если этот материал присутствует в BOM этого заказа
- **FR-009**: Система MUST блокировать списание, если запрашиваемое количество превышает текущий остаток материала на складе; в ошибке указывается доступный остаток
- **FR-010**: Система MUST разрешать частичное списание — несколько списаний по одной строке BOM суммируются в фактический расход
- **FR-011**: Система MUST показывать предупреждение (но не блокировать) при списании, если суммарный фактический расход по строке BOM превысит требуемое по BOM количество
- **FR-012**: Система MUST блокировать списание для заказов в статусах «Завершён» и «Отменён»
- **FR-013**: Каждая операция списания MUST сохраняться как отдельная запись в журнале движений с типом «Списание», ссылкой на заказ, количеством, комментарием, актором и датой/временем
- **FR-014**: Текущий остаток материала на складе MUST вычисляться как сумма приходов минус сумма списаний (по этому материалу, без фильтрации по заказу)

#### Расход по заказу

- **FR-015**: Система MUST предоставлять агрегированное представление расхода материалов по заказу: для каждой строки BOM — материал, ед.изм., требуется (BOM), списано (сумма всех списаний по этому материалу+заказу), остаток к списанию = max(0, требуется − списано)
- **FR-016**: Система MUST визуально маркировать строки расхода с перерасходом (списано > требуется) и показывать величину превышения

#### RBAC

- **FR-017**: Система MUST разрешать редактирование BOM (добавление, редактирование, удаление строк) пользователям с ролью `ORDER_MANAGER` или `ADMIN`
- **FR-018**: Система MUST разрешать выполнение операции списания пользователям с ролью `WAREHOUSE` или `ADMIN`
- **FR-019**: Система MUST разрешать просмотр BOM и расхода по заказу следующим ролям: `ORDER_MANAGER`, `WAREHOUSE`, `PRODUCTION_SUPERVISOR`, `ADMIN`. Роль `PRODUCTION_EXECUTOR` НЕ имеет доступа к этим секциям карточки заказа.

#### Аудит

- **FR-020**: Система MUST фиксировать в журнале аудита (категория «Склад») все события: BOM_LINE_ADDED, BOM_LINE_UPDATED, BOM_LINE_REMOVED, STOCK_CONSUMPTION — с привязкой к актору, заказу и материалу

#### Frontend

- **FR-021**: Система MUST отображать в карточке заказа (`/cabinet/orders/{id}`) две новые секции: «Материалы (BOM)» (таблица + кнопка «Добавить материал») и «Расход материалов» (агрегированная таблица)
- **FR-022**: Система MUST поддерживать оформление списания из карточки заказа (preselected заказ, выбор материала из BOM этого заказа) — это основной UX-вход
- **FR-023**: Система MUST дополнительно поддерживать оформление списания из раздела «Склад» (выбор заказа из выпадающего списка активных заказов, далее выбор материала из BOM выбранного заказа) — альтернативный вход для кладовщика
- **FR-024**: Список заказов, доступных для списания, MUST включать только заказы в статусах, разрешающих списание (не «Завершён» и не «Отменён»), и только заказы с непустым BOM

### Key Entities

- **Строка спецификации материала заказа (BOM line / OrderMaterialRequirement)**: связь заказ↔материал с требуемым количеством. Атрибуты: ссылка на заказ, ссылка на материал, требуемое количество (положительный BigDecimal), опциональный комментарий, даты создания и обновления. Уникальная пара (заказ, материал).
- **Движение материала (StockMovement)** *(расширение из 011)*: добавляется новый тип «Списание» (CONSUMPTION) и опциональная ссылка на заказ. Для приходов ссылка на заказ — пустая, для списаний — обязательная.
- **Расход по заказу (агрегат, не отдельная сущность)**: вычисляемое представление = BOM ⨝ суммированные списания по (заказ, материал).

### Constitution Alignment

- **ERP/domain fit**: Закрывает сценарий «заказ → материалы → производство». Это ключевая часть Phase 2 ERP-функциональности: связь между складом и операционным контуром заказов.
- **TOC readiness**: Расход материалов по заказу — индикатор фактической обеспеченности производства. Будущие спеки (резервирование, учёт по этапам) надстраиваются поверх этой модели. Перерасход по BOM фиксируется и виден — это сигнал для анализа узких мест.
- **Traceability/audit**: Каждое движение материала под заказ хранит актора, timestamp и ссылку на заказ. Изменения BOM записываются в аудит-лог. Восстановление истории «куда ушёл материал X» возможно по полю order_id в журнале движений.
- **Security/API boundary**: Разделение ролей: `ORDER_MANAGER` редактирует BOM (планирование), `WAREHOUSE` выполняет списание (физическая операция). Просмотр доступен ролям `ADMIN`, `ORDER_MANAGER`, `WAREHOUSE`, `PRODUCTION_SUPERVISOR`. JWT-аутентификация без изменений.

## Success Criteria

### Measurable Outcomes

- **SC-001**: Менеджер заказа может добавить строку BOM в существующий заказ менее чем за 30 секунд
- **SC-002**: Кладовщик может оформить списание материала по заказу из карточки заказа менее чем за 30 секунд (от открытия карточки до подтверждения операции)
- **SC-003**: При попытке списать материал в количестве, превышающем остаток, операция блокируется в 100% случаев и пользователь видит понятную ошибку с указанием доступного остатка
- **SC-004**: Открыв карточку заказа, пользователь в течение 5 секунд понимает, какие материалы по заказу полностью выданы, какие частично, и где есть перерасход (по визуальным маркерам секции «Расход материалов»)
- **SC-005**: Все операции BOM и списания на 100% попадают в журнал аудита (нет пропусков); запись содержит достаточно данных для понимания, что произошло, без необходимости открывать саму карточку заказа
- **SC-006**: Текущий остаток материала на складе всегда корректен: для любой выборки приходов и списаний по материалу остаток = Σ приходов − Σ списаний

## Assumptions

- Терминология: операция называется «Списание» (русский UI), backend-тип — `CONSUMPTION` (по аналогии с `RECEIPT` из 011). Слово «WRITE_OFF» не используется во избежание путаницы с бухгалтерским «списанием убытков».
- BOM редактирует Order Manager (он создаёт и ведёт заказ, он же планирует требуемые материалы). Production Supervisor как соредактор не входит в эту спеку — его роль ограничивается просмотром.
- Списание привязывается к заказу, не к производственной задаче (production task). Учёт по этапам через task откладывается на будущую спеку.
- Перерасход по BOM (списано больше, чем требуется) — допустимое событие, отображается как warning, не блокируется. На практике перерасход случается из-за брака и неточностей в плане; блокировать его — мешать реальной работе.
- Возврат материала на склад (например, при отмене заказа с уже сделанными списаниями) **в эту спеку не входит** и будет рассмотрен отдельно. Отмена заказа со списаниями оставляет фактический расход в журнале как есть.
- Резервирование материала под BOM (без физического списания) **не входит** в эту спеку.
- Уведомления о перерасходе материала по BOM (через модуль notifications спек 008–010) **не входят** в эту спеку. Перерасход виден только в UI карточки заказа (US3) и в журнале аудита.
- Шаблоны BOM по типам продукции **не входят**: BOM создаётся вручную для каждого заказа с нуля.
- Списание возможно только по тем материалам, которые есть в BOM выбранного заказа. Это сознательное ограничение, обеспечивающее плановость расхода.
- Один материал = одна строка BOM на заказ (уникальность пары). Если требуется несколько порций одного материала на разных этапах — пользователь складывает их в одну суммарную строку.
- Статусы заказа, при которых разрешено списание и редактирование BOM, наследуются из существующей модели заказов (`orders/domain`). Запрещены: «Завершён» и «Отменён». Все остальные статусы (черновик, в работе и т.д.) — разрешены.
- API endpoints размещаются в существующем модуле inventory: BOM по пути `/api/orders/{orderId}/bom`, списание — `POST /api/materials/{id}/consume`, агрегат расхода — `GET /api/orders/{orderId}/material-usage`.
- Cross-module доступ inventory → orders реализуется через новый порт `OrderLookupPort` (минимум — проверка существования заказа и получение его статуса), по образцу `ProductionOrderSourcePort` из модуля production.
- Frontend-страница списания материалов из раздела «Склад» (альтернативный вход) — это та же модалка, что и из карточки заказа, но с дополнительным шагом выбора заказа. UI-компонент модалки списания общий для обеих точек входа.
