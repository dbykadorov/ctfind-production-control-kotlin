# Quickstart: BOM + Списание материалов

Ручная проверка фичи 012 после реализации. Подразумевается, что бэкенд и фронт собраны, миграции применены, Docker-стек запущен.

## Предусловия

```sh
make docker-reset && make docker-up-detached && make health
# → {"status":"UP"}
```

В seed-данных должны быть:
- Пользователь `admin` с ролью `ADMIN` (стандарт из спеки 002).
- Тестовый Order Manager (создаётся вручную через UI или seed-runner).
- Тестовый Warehouse-кладовщик с ролью `WAREHOUSE`.
- Хотя бы один заказ в статусе `NEW` или `IN_WORK` для прогона.

Если seed нет, авторизуйтесь как `admin` и создайте пользователей: cabinet → Журнал действий пока не открывает CRUD пользователей в этой спеке, выполняется через прямой API-вызов или через ранее существующее средство наполнения.

## Сценарий 1 — BOM (US1)

1. Залогиниться как Order Manager.
2. Создать материал «Фанера 10мм» (ед.изм. SQUARE_METER) через `/cabinet/warehouse` → «Добавить материал».
3. Залогиниться как WAREHOUSE-кладовщик, оформить приход 50 м² фанеры через карточку материала.
4. Залогиниться обратно как Order Manager.
5. Открыть карточку существующего заказа `/cabinet/orders/<id>`.
6. **Проверка**: видна секция «Материалы (BOM)» с пустой таблицей и кнопкой «Добавить материал».
7. Нажать «Добавить материал» → выбрать «Фанера 10мм» → ввести `10` → комментарий «На основу» → сохранить.
8. **Проверка**: в таблице BOM появилась строка `Фанера 10мм / 10 м² / На основу`.
9. Попробовать ещё раз добавить «Фанера 10мм` → **ожидается ошибка** «Этот материал уже есть в спецификации заказа».
10. Кликнуть «Редактировать» на строке → изменить количество на `12` → сохранить.
11. **Проверка**: количество в таблице обновилось.
12. Кликнуть «Удалить» → подтвердить → строка исчезла.
13. Снова добавить «Фанера 10мм / 10 / на основу» (понадобится для следующего сценария).

## Сценарий 2 — Списание из карточки заказа (US2 основной)

1. Залогиниться как WAREHOUSE.
2. Открыть карточку того же заказа.
3. **Проверка**: секции «Материалы (BOM)» и «Расход материалов» отображаются. Возможность «Добавить материал» **скрыта или disabled** (нет роли Order Manager).
4. В секции «Расход материалов» видна строка `Фанера 10мм / требуется 10 / списано 0 / остаток 10`. Видна кнопка «Списать» (роль WAREHOUSE).
5. Нажать «Списать» → диалог. Заказ преселектирован, в выпадающем списке материалов **только** «Фанера 10мм». Ввести `5`, комментарий «Выдано в цех» → сохранить.
6. **Проверка**: после успеха диалог закрывается, таблица обновилась: `требуется 10 / списано 5 / остаток 5`. Текущий остаток фанеры на странице склада уменьшился с 50 до 45.
7. Списать ещё `4` → таблица `требуется 10 / списано 9 / остаток 1`.
8. Попытаться списать `100` → **ожидается ошибка** «Недостаточный остаток на складе. Доступно: 41 м²».
9. Списать ещё `5` (всего станет 14, перерасход +4):
   - В диалоге появляется warning «Перерасход относительно BOM: +4 м²» (не блокирующий).
   - После подтверждения операция сохраняется. В таблице расхода: `требуется 10 / списано 14 / остаток 0` + индикатор перерасхода `+4`.

## Сценарий 3 — Списание из раздела «Склад» (US2 альтернативный)

1. Не выходя из роли WAREHOUSE, перейти на `/cabinet/warehouse`.
2. **Проверка**: рядом с кнопкой «Добавить материал» (или в шапке) есть кнопка «Списать».
3. Нажать «Списать». Диалог показывает шаг выбора заказа: autocomplete по orderNumber/customerName.
4. Начать вводить номер заказа из сценария 2 → в выпадашке появляется заказ.
5. Выбрать → диалог переходит к выбору материала из BOM этого заказа. «Фанера 10мм» доступна.
6. Списать ещё `1` → успех.
7. Открыть `/cabinet/orders/<id>` → таблица расхода обновлена.

## Сценарий 4 — Lifecycle (статус заказа SHIPPED)

1. Перевести заказ через workflow в статус `READY` → `SHIPPED` (под Order Manager).
2. Залогиниться как Order Manager и открыть карточку.
3. **Проверка**: секция BOM видна, но кнопки «Добавить / Редактировать / Удалить» disabled (или скрыты).
4. Залогиниться как WAREHOUSE и открыть ту же карточку.
5. **Проверка**: кнопка «Списать» в секции расхода disabled. Заказ не появляется в picker списания со страницы склада.

## Сценарий 5 — Блокировка удаления BOM-строки и материала

1. Создать новый заказ. Добавить в его BOM «Фанера 10мм» → списать `1`.
2. Под Order Manager попробовать удалить эту BOM-строку → **ожидается ошибка** «Невозможно удалить материал, по которому уже были списания».
3. Под Order Manager попробовать удалить материал «Фанера 10мм» через `/cabinet/warehouse` → **ожидается ошибка** удаления (есть движения).
4. Создать ещё материал «Скоба» (без приходов и движений). Добавить его в BOM нового заказа (без списаний). Попробовать удалить материал → **ожидается ошибка**: «Материал используется в BOM активного заказа» (правило Q3).
5. Перевести заказ в `SHIPPED`. Снова попытаться удалить материал «Скоба» → **успех** (нет движений, нет активных BOM-ссылок).

## Сценарий 6 — Аудит

1. Залогиниться как `admin`.
2. Открыть `/cabinet/audit` (журнал действий).
3. Применить фильтр категории `INVENTORY`.
4. **Проверка**: видны записи с типами `BOM_LINE_ADDED`, `BOM_LINE_UPDATED`, `BOM_LINE_REMOVED`, `STOCK_CONSUMPTION`. Каждая запись содержит актора, заказ (в summary или metadata), материал и количество.

## Verification commands

```sh
# Backend
make backend-test-docker

# Frontend
cd frontend/cabinet && pnpm typecheck && pnpm vitest run && pnpm build

# Health check after redeploy
make docker-up-detached && make health
```

Все три должны проходить чисто перед тем, как фича считается завершённой.

## Известные сценарии негативных тестов (intgr/unit)

Список не исчерпывающий, для напоминания авторам тестов:
- BOM `quantity = 0` → 400 `VALIDATION_FAILED`.
- BOM `comment` 501 символ → 400.
- Параллельно два consume по 30 при остатке 50 → один успех, один `INSUFFICIENT_STOCK`.
- Удалить материал, использованный в BOM активного заказа → отказ.
- `GET /api/orders/active-for-consumption` от пользователя без `WAREHOUSE` → 403.
- Списание под `SHIPPED` заказ → 409 `ORDER_LOCKED`.
- Списание материала, отсутствующего в BOM → 409 `MATERIAL_NOT_IN_BOM`.
