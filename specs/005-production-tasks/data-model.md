# Data Model: Production Tasks

## ProductionTask

Represents one executable production work item created from a customer order and, when applicable, a specific order item.

**Fields**:

- `id: UUID`
- `taskNumber: String` - unique human-readable production task number generated by the backend
- `orderId: UUID` - source customer order
- `orderItemId: UUID?` - source order item when the task is tied to a specific line
- `purpose: String` - required user-visible reason for the work, especially when multiple tasks exist for the same order item
- `itemName: String` - display copy from source order item or user-provided task item context
- `quantity: BigDecimal`
- `uom: String`
- `status: ProductionTaskStatus`
- `previousActiveStatus: ProductionTaskStatus?` - stores `NOT_STARTED` or `IN_PROGRESS` while task is blocked
- `executorUserId: UUID?` - exactly one current executor assignee; nullable until planned/assigned
- `plannedStartDate: LocalDate?`
- `plannedFinishDate: LocalDate?`
- `blockedReason: String?`
- `createdByUserId: UUID`
- `createdAt: Instant`
- `updatedAt: Instant`
- `version: Long` - optimistic concurrency token

**Validation**:

- `taskNumber` is required, unique, and generated server-side.
- `orderId` must reference an existing source order.
- `orderItemId`, when present, must belong to `orderId`.
- `purpose` must be non-blank and bounded to a reasonable UI length.
- Multiple tasks may reference the same `orderItemId` only when each task has a distinct purpose.
- `quantity` must be greater than zero.
- `uom` must be non-blank.
- New tasks always start as `NOT_STARTED`.
- `executorUserId`, when present, must identify an active executor-capable user.
- `plannedFinishDate` must not be earlier than `plannedStartDate` when both are provided.
- Status, assignment, and planning updates must include the expected `version`.
- Completed tasks reject regular assignment/planning/status edits except read-only review.

**Relationships**:

- Many production tasks can belong to one customer order.
- Many production tasks can reference one customer order item if each has a distinct purpose.
- One production task has zero or more history events.
- One production task has zero or more audit events.

## ProductionTaskStatus

Lifecycle state for a production task.

**Values**:

- `NOT_STARTED` - user-facing label: `не начато`
- `IN_PROGRESS` - user-facing label: `в работе`
- `BLOCKED` - user-facing label: `заблокировано`
- `COMPLETED` - user-facing label: `выполнено`

**Allowed transitions**:

```text
NOT_STARTED -> IN_PROGRESS -> COMPLETED
NOT_STARTED -> BLOCKED -> NOT_STARTED
IN_PROGRESS -> BLOCKED -> IN_PROGRESS
```

**Forbidden transitions**:

- Reverse transitions except unblock returning to the previous active status.
- Skipping directly from `NOT_STARTED` to `COMPLETED`.
- Reapplying the current status as a business transition.
- Completing a blocked task without first unblocking it.
- Any executor status update for a task not assigned to that executor.

## ProductionTaskAssignment

Represents the current responsibility for a task.

**Fields**:

- `taskId: UUID`
- `executorUserId: UUID`
- `assignedByUserId: UUID`
- `assignedAt: Instant`

**Validation**:

- Exactly one current executor may be assigned to a task.
- Reassignment is allowed before completion by `ADMIN`, `ORDER_MANAGER`, or `PRODUCTION_SUPERVISOR`.
- Reassignment creates a history event and audit event.
- Work area, team, and multi-executor assignment are out of scope for Feature 005.

## ProductionTaskHistoryEvent

Represents one traceable business event in a production task timeline.

**Fields**:

- `id: UUID`
- `taskId: UUID`
- `eventType: ProductionTaskHistoryEventType`
- `actorUserId: UUID`
- `eventAt: Instant`
- `fromStatus: ProductionTaskStatus?`
- `toStatus: ProductionTaskStatus?`
- `previousExecutorUserId: UUID?`
- `newExecutorUserId: UUID?`
- `plannedStartDateBefore: LocalDate?`
- `plannedStartDateAfter: LocalDate?`
- `plannedFinishDateBefore: LocalDate?`
- `plannedFinishDateAfter: LocalDate?`
- `reason: String?` - required for block events
- `note: String?`

**Event types**:

- `CREATED`
- `ASSIGNED`
- `PLANNING_UPDATED`
- `STATUS_CHANGED`
- `BLOCKED`
- `UNBLOCKED`
- `COMPLETED`

**Validation**:

- Every successful task creation creates a `CREATED` event.
- Every successful assignment/reassignment creates an `ASSIGNED` event.
- Every successful planned-date change creates a `PLANNING_UPDATED` event.
- Every successful lifecycle transition creates a status-related event.
- Block events require `reason`.
- Events are append-only after creation.

## ProductionTaskAuditEvent

Business audit record for significant production task actions. This may reuse the existing audit storage pattern when implemented.

**Fields**:

- `id: UUID`
- `eventType: String` - e.g. `PRODUCTION_TASK_CREATED`, `PRODUCTION_TASK_ASSIGNED`, `PRODUCTION_TASK_STATUS_CHANGED`
- `actorUserId: UUID`
- `targetType: String` - `PRODUCTION_TASK`
- `targetId: UUID`
- `eventAt: Instant`
- `summary: String`
- `metadata: Json?`

**Validation**:

- Every successful create, assign, planning update, status change, block, unblock, and completion creates an audit event.
- Failed validation, forbidden writes, invalid transitions, and stale writes must not mutate task data.

## Query Models

### ProductionTaskListFilter

- `search: String?` - matches task number, order number, customer display name, item name, or purpose
- `status: ProductionTaskStatus?`
- `orderId: UUID?`
- `orderItemId: UUID?`
- `executorUserId: UUID?`
- `assignedToMe: Boolean?`
- `blockedOnly: Boolean?`
- `activeOnly: Boolean?` - non-completed tasks
- `dueDateFrom: LocalDate?`
- `dueDateTo: LocalDate?`
- `page: Int`
- `size: Int`
- `sort: String?`

### ProductionTaskDetail

- Full `ProductionTask`
- Source order summary: order id, order number, customer display name, order status, delivery date
- Source order item summary when applicable
- Executor display name when assigned
- History timeline
- `allowedActions` for the current user

## State And Permission Rules

- `ADMIN` can create, assign, plan, view, and update any production task.
- `ORDER_MANAGER` can create tasks from orders and assign/plan production tasks.
- `PRODUCTION_SUPERVISOR` can view all tasks, assign/plan tasks, and update any task status.
- `PRODUCTION_EXECUTOR` can view only tasks assigned to them and update status only for those assigned tasks.
- Other authenticated read-only Phase 1 roles can view only tasks allowed by their visibility scope and cannot mutate tasks.
- Unauthenticated requests return `401`.
- Authenticated users without permission receive `403` for forbidden list/detail visibility or mutation attempts.
- Stale updates return `409` and leave data unchanged.
- Invalid validation or lifecycle transitions return `400` or `422` and leave data unchanged.
