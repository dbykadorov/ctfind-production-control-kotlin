# Руководство администратора (Phase 01)

Документ описывает практическую работу роли `ADMIN` в кабинете Phase 01: вход, навигацию, доступные операции и базовые проверки.

## 1. Назначение роли

Администратор отвечает за:

- первичный доступ к системе (bootstrap суперадмина);
- управление пользователями и ролями в кабинете;
- доступ к журналу аудита;
- контроль ролей и доступа к рабочим разделам.

## 2. Вход и старт

### Локальное окружение (`local`)

- При старте приложения выполняется локальный seed администратора.
- Можно войти под локальной учетной записью администратора, после чего открыть кабинет.

### Production-like окружения

- Если в системе еще нет пользователя с ролью `ADMIN`, приложение ожидает bootstrap-параметры:
  - `APP_SUPERADMIN_LOGIN`
  - `APP_SUPERADMIN_DISPLAY_NAME`
  - `APP_SUPERADMIN_PASSWORD`
- При отсутствии обязательных параметров приложение завершает запуск с операторской ошибкой конфигурации.
- Повторные перезапуски не создают дубликаты администратора.

## 3. Навигация администратора

Администратор видит все основные разделы Phase 01, включая:

- `Обзор`
- `Заказы`
- `Производство` / `Доска`
- `Клиенты`
- `Склад`
- `Журнал действий`
- `Пользователи`
- `Уведомления`

Маршрут управления пользователями: `/cabinet/users`.

## 4. Раздел "Пользователи"

Раздел доступен только роли `ADMIN`.

### Что доступно

- Просмотр списка пользователей.
- Поиск по логину или отображаемому имени.
- Просмотр назначенных ролей у каждого пользователя.
- Создание пользователя с выбором одной или нескольких ролей.

### Создание пользователя

Форма создания включает:

- логин;
- отображаемое имя;
- начальный пароль;
- роли (одна или несколько).

Поддерживаются роль-коды Phase 01:

- `ADMIN`
- `ORDER_MANAGER`
- `WAREHOUSE`
- `PRODUCTION_SUPERVISOR`
- `PRODUCTION_EXECUTOR`

### Ограничения и валидация

- Логин должен быть непустым и уникальным.
- Отображаемое имя должно быть непустым.
- Начальный пароль должен быть непустым.
- Нельзя создать пользователя без ролей.
- Нельзя указать неизвестные role-коды.
- Пароль не возвращается в API-ответе и не отображается в UI.

## 5. Быстрый чек-лист проверки ADMIN

1. Войти под администратором.
2. Открыть `/cabinet/users`.
3. Создать `warehouse.demo` с ролью `WAREHOUSE`.
4. Убедиться, что пользователь появился в списке и в поиске.
5. Выйти и войти под `warehouse.demo`.
6. Проверить, что доступны только разрешенные разделы роли.
7. Проверить, что `warehouse.demo` не видит `Журнал действий` и `Пользователи`.

## 6. Ограничения Phase 01

В текущем инкременте не поддерживаются:

- self-registration;
- восстановление/смена пароля;
- редактирование, блокировка и удаление пользователей;
- массовый импорт пользователей;
- внешние IdP/LDAP/OAuth интеграции.
